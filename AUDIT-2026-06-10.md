## 1. Executive summary

LaserForge 2.0 is a structurally disciplined codebase — strict TypeScript compiles clean, 819 tests across 105 files pass, and the perceptual trace harness meets or exceeds its pinned IoU baselines. The architecture (pure core, module boundaries, discriminated unions, deterministic G-code) is genuinely enforced in most places and is the project's strongest asset.

But the audit found that **the project's central promise — output fidelity — is broken on the default configuration**, and that the test suite is structurally incapable of seeing it. The flagship defect: **every image-mode engrave is emitted vertically mirrored on the default front-left origin** (C2), while the WYSIWYG preview deliberately renders in scene space and therefore looks correct. Two further criticals: **no laser-off path exists on disconnect/tab-close mid-job** (C3, M3 constant-power beam parked on the workpiece — fire-adjacent), and the audit-validity finding above (C1).

Every critical/high finding survived two independent adversarial refuters; every medium survived one. **Zero findings were killed outright**; one (the stash's Pause/Stop failure path, HD1) is genuinely disputed. "Unverified" findings are all low-severity and were not adversarially checked.

**Top 5 risks**
1. **Stale audit target / unguarded main** (C1): the snapshot is 88 commits behind, and red-CI commits were pushed straight to main — no branch protection.
2. **Mirrored raster output on the default origin** (C2): silent, 100%-reproducible wrong physical burns; masked by both preview and tests.
3. **Laser-off guarantee has holes** (C3, H5, H6, H13): disconnect/tab-close leaves an M3 beam energized; GRBL `error:N` is silently swallowed mid-stream; jog/home mid-job corrupts RX accounting; native dialogs freeze the Stop button.
4. **Import fidelity** (H4, H8, H9): SVG transforms ignored, compact arc flags dropped, no unit conversion — common real-world files import wrong.
5. **Potrace GPL provenance** (H11): the default trace path now runs an unattributed in-house potrace port carrying GPL-source fingerprints, with no ADR, in a proprietary repo.

**Top 5 opportunities**
1. **Centerline trace already shipped** — LightBurn has no equivalent; it needs a perceptual metric, then it's a headline differentiator.
2. **Firmware-true auto-config + planner-true ETA** — `$$` ingestion and the junction-deviation estimator are already ahead of LightBurn; add the missing raster term (P0 parity gap).
3. **The zero-install web wedge** — LightBurn orphaned Linux in 2024 and has renewal friction; nobody else is browser-native. (The "PWA-installable" claim is currently unimplemented.)
4. **Power-true WYSIWYG preview + deterministic G-code** — provable in CI, marketable against LightBurn's "fake preview" and update-regression history.
5. **Material test / presets / console / primitives** — the four table-stakes gaps that keep LightBurn installed on users' machines.

#### C2. Raster engrave is emitted vertically mirrored on front-\* origins (the default) and X-mirrored on \*-right origins — pixel S-values land at flipped coordinates

*(Merged: two independently-filed reports of the same root defect, one at the emit layer, one at the compile layer; 4 adversarial verifications total, all confirming.)*

- **Location:** `src/core/job/compile-job.ts:88` (`rasterBoundsInMachineCoords`, lines 138–156) and `src/core/raster/emit-raster.ts:103`
- **Status:** **confirmed** (structurally — full pipeline traced end-to-end by three auditors; NOT hardware-verified)
- **Description:** Vector points are origin-transformed individually (`toMachineCoords`: front-left returns `{x, y: bedH - p.y}`), but the raster path collapses the origin transform to an axis-aligned AABB (min/max of the four transformed corners), which destroys axis orientation. The luma buffer is row-major top-down (canvas `getImageData`; row 0 = scene top), and `emit-raster.ts:103` burns row 0 at `worldY = bounds.minY + 0.5·pixelHeightMm` — the **smallest** machine Y. On the default `front-left` origin (the shipped `DEFAULT_DEVICE_PROFILE`), machine minY corresponds to the image's scene-**bottom** edge — so every image engrave burns **vertically mirrored** about its own midline, out of register with coincident vector layers (which transform correctly per point). `*-right` origins mirror columns; `center` flips Y; only `rear-left` (identity) is correct. Nothing catches it: no test pins luma-row-to-machine-Y orientation; `draw-raster-preview.ts` deliberately renders in scene space (ADR-028: "The machine Y-flip stays confined to the G-code path"), so **WYSIWYG looks correct while the emitted G-code is mirrored**; the F-F2 hardware checklist has no orientation item.
- **Evidence:** `emit-raster.ts:103`: `const worldY = input.bounds.minY + (y + 0.5) * pixelHeightMm;` — `origin-transform.ts:27-29`: `case 'front-left': … return { x: p.x, y: bedH - p.y };` — `device-profile.ts:82`: `origin: 'front-left'` — `compile-job.ts:138-156` returns only `{minX,maxX,minY,maxY}` of the transformed corners; no row/column reorder exists anywhere downstream (greps verified).
- **LightBurn:** Engraves a bitmap exactly as placed and displayed in the workspace, for every machine-origin setting; image and vector layers of the same artwork stay in register. A mirrored raster vs its own trace overlay would never occur.
- **Suggested fix:** In `compileRasterGroup`, after computing the machine AABB, reverse luma row order when the origin transform negates Y (front-left, front-right, center) and reverse column order when it negates X (front-right, rear-right). Pin with a compile-job test (asymmetric 2-row image, dark row must land at the high machine Y on front-left), an emit-level pin, and a perceptual fixture comparing raster vs vector emit. Then verify on the F.2.f hardware burn with an asymmetric test image (e.g. text). **Re-check against upstream emit-raster.ts (rewritten) first.**

#### C3. In-app Disconnect (and tab close, and renderer crash) mid-job never sends a stop — the laser can remain energized at a standstill

- **Location:** `src/ui/state/laser-store.ts:303` (`disconnect()`)
- **Status:** **confirmed** (2 refuters; code-path analysis + documented GRBL 1.1 laser-mode semantics; NOT hardware-verified)
- **Description:** `disconnect()` tears down subscriptions and closes the port without sending `RT_SOFT_RESET` (0x18) or `M5`. The Disconnect button is enabled during an active stream (no streaming gate in `ConnectionBar`), and there is **no** beforeunload/pagehide handler that stops the machine (the only beforeunload handler is autosave). The web adapter's `close()` writes no bytes. Line/Fill jobs run under `M3` (preamble `M3 S0`); in GRBL 1.1 laser mode, M3 constant-power keeps the beam ON at the programmed S even when motion stops. So Disconnect / tab close / renderer crash mid-cut lets GRBL drain its RX + planner buffers and then sit Idle with a **stationary beam burning one spot indefinitely**, unless the particular board happens to reset on port close (DTR behavior is board-dependent and unverified for the Falcon A1 Pro). The only live laser-off command the app ever sends is `RT_SOFT_RESET` inside `stopJob`. Raster (M4) groups fail safe; M3 cut/fill groups do not. PROJECT.md non-negotiable #9 (E-stop reachable from any window state) is undermined: the most reachable "stop" — closing the window — leaves the beam on.
- **Evidence:** `laser-store.ts:303-316`: `disconnect: async () => { const conn = refs.connection; teardown(); if (conn !== null) await conn.close()…` — no reset/M5. Grep `M5|SOFT_RESET` in src/ui → only `stopJob` at `laser-store.ts:404`. Grep `beforeunload` → only `use-autosave.ts`.
- **LightBurn:** Treats an active job as something that must be stopped, not abandoned; its Stop aborts the stream and resets the controller. (Exact LightBurn disconnect wire sequence not verified; the contrast claimed is that LaserForge has NO laser-off path on any abandonment event.)
- **Suggested fix:** In `disconnect()`: if the streamer is streaming/paused, best-effort write `RT_SOFT_RESET` before close and mark the streamer cancelled. Disable (or confirm + auto-stop) Disconnect while a job is active. Add a beforeunload/pagehide handler that synchronously writes 0x18 when a job is active. Consider M5 after soft reset belt-and-braces.

#### H1. Custom-origin (G92) Start is preflight-blocked for every fill and image job — overscan lead-ins go negative after user-origin translation and are checked against the machine-frame bed rectangle

- **Location:** `src/ui/laser/start-job-readiness.ts:47`; root cause spans `job-origin.ts:34-40`, `fill-overscan.ts:27`, `emit-raster.ts:168`, `predicates.ts:86`
- **Status:** **confirmed** (2 refuters; statically traced, not executed)
- **Description:** With a custom origin active, `applyJobOrigin` anchors the burn-bounds front-left corner at work (0,0) — overscan deliberately excluded from job bounds. At emit time, fill lead-ins extend 5 mm (default) past the burn span and raster rows rapid to `activeStartX − 5`, producing `G0 X-5.000 … S0`. Preflight then checks these WORK coordinates against `[0,bed]` and fails with "X out of bed: −5" — so the canonical diode workflow (G92 origin at material corner + Fill or Image mode) is **hard-blocked with a misleading error**, even when the physical runway is fine. The complementary gap: the physical check (`findOriginBoundsIssue`) uses bounds **without** overscan, so a runway that really leaves the bed near the physical edge is NOT caught. Both halves stem from checking overscan in the wrong coordinate frame. No test covers fill/raster + user-origin through preflight. ("Every" is slightly overbroad — fillOverscanMm=0 is a workaround for Fill; Image has none.)
- **Evidence:** `job-bounds.ts:60-62`: "Overscan is intentionally NOT included…" — `grbl-strategy.ts:95`: `G0 X${fmt(run.leadStart.x)} … S0` with leadStart = burnStart − 5 mm — `predicates.ts:86`: `if (x !== null && (x < 0 || x > bed.width))`.
- **LightBurn:** "Start From: User Origin / Current Position" validates extents (including overscan) against the working area using the actual origin position; a mid-bed user-origin fill/image job starts normally.
- **Suggested fix:** Run the bounds preflight for user-origin jobs in the physical frame (offset by `wcoCache`, failing closed with a frame-aware message when WCO is unknown), or exclude the work-frame `[0,bed]` check for user-origin jobs and extend `findOriginBoundsIssue` to include the overscan runway. Regression test: fill layer + default overscan + user-origin placement must not be rejected when the job physically fits.

#### H2. Fill hatch spacing and angle are computed in object-local coordinates before the transform — emitted spacing scales with the object; angle rotates/mirrors with object and origin flip

- **Location:** `src/core/job/compile-job.ts:229` (hatching before `applyTransform`/`toMachineCoords`)
- **Status:** **confirmed** (2 refuters; statically traced, not perceptually verified)
- **Description:** `fillHatching` runs on raw object-local polylines; scale/rotate/flip are applied per point afterwards. So `hatchSpacingMm` is honored in OBJECT units, not bed mm: a fit-to-bed-scaled import (scale 0.36) burns hatch at 0.36× the configured spacing (~3× energy density, char risk); a scaled-up object under-fills. Worse than originally reported: **every traced image carries scale ≈ 0.265** (px→mm), and trace layers default to fill — so the standard trace→fill flow always burns ~3.8× denser than configured, defeating the `MIN_HATCH_SPACING_MM` safety clamp (applied pre-scale). `hatchAngleDeg` rotates with `rotationDeg` and the front-left Y-flip mirrors the machine-frame angle. ADR-019 records the pre-transform mechanism but not the unit/angle consequence; PROJECT.md non-negotiable #6 promises mm-honest units. The preview reuses the same function, so it is self-consistent and hides the divergence.
- **Evidence:** `compile-job.ts:226-235`: hatching first, `applyTransform`+`toMachineCoords` after — `fill-hatching.ts:53`: spacing used raw in the local frame — `fit-to-bed.ts:31` stores scale on the transform.
- **LightBurn:** Line Interval is an absolute machine-space distance (mm); Scan Angle is machine-frame degrees. Scaling/rotating/mirroring a shape never changes the burned interval or angle.
- **Suggested fix:** Hatch in scene/machine space: transform contours first, then hatch with the layer's mm spacing/angle (angle corrected for origin flip), then map through `toMachineCoords` only. Pin: scaleX=scaleY=2 + hatchSpacingMm=1 must emit lines 1 mm apart.

#### H3. Preview mode draws the vector toolpath in machine coordinates on the scene-space canvas — mirrored about the bed midline on the default origin, and inconsistent with the scene-space raster sim

- **Location:** `src/ui/workspace/draw-preview.ts:79` (`buildPreviewToolpath`)

#### H4. SVG `transform` attributes (element and group) are completely ignored at import — Inkscape/Illustrator exports import with misplaced/mis-scaled geometry

- **Location:** `src/io/svg/parse-svg.ts:112` (`walkGeometry`)
- **Status:** **confirmed** (2 refuters; structural — no Inkscape fixture executed, but the code admits no path by which a transform could apply)
- **Description:** `walkGeometry` iterates `querySelectorAll('*')` flat and converts each primitive from its own geometry attributes only. Neither parse-svg nor shape-to-polylines nor parse-path-d ever reads the `transform` attribute, and there is no ancestor `<g>` accumulation (case-insensitive grep over src/io/svg matches only `IDENTITY_TRANSFORM`, an unrelated scene placement). Real exporters rely heavily on transforms (Inkscape layer `<g transform="translate(...)">`, `matrix(...)` on elements); such files import with every transformed element at raw untransformed coordinates. Not flagged in any doc — unlike other admitted Phase A gaps. Bounds come from the viewBox, masking the problem until geometry visibly lands wrong.
- **Evidence:** `parse-svg.ts:117-135` reads only `stroke` plus geometry attributes; `shape-to-polylines.ts:16-132` reads only d/x1/y1/points/x/y/width/height/cx/cy/r/rx/ry.
- **LightBurn:** Applies the full SVG transform stack on import; an Inkscape export looks identical in LightBurn's workspace.
- **Suggested fix:** Parse `transform` lists (translate/scale/rotate/matrix/skew per SVG 1.1) into 2×3 matrices, accumulate ancestor→element during a recursive descent (replacing the flat `querySelectorAll`), map every emitted point. Add an Inkscape-style fixture test.

#### H5. GRBL `error:N` mid-stream is swallowed: the job continues past rejected lines (wrong-geometry burn risk), reports a clean finish, and the error is never surfaced to the operator

*(Merged: two reports of the same defect — stream-control policy in the core streamer, and the never-read `lastError` in the UI; 3 adversarial verifications, all confirming.)*

- **Location:** `src/core/controllers/grbl/streamer.ts:140` (`onAck`) and `src/ui/state/laser-line-handler.ts:145`
- **Status:** **confirmed** (structurally; not reproduced on hardware)
- **Description:** `onAck` treats `error` identically to `ok` for stream control — only `alarm` cancels. `handleLine` sets `lastError` and keeps streaming. An `error:N` means GRBL **rejected and did not execute** the line: a rejected `G0 … S0` reposition (serial corruption — common on diode USB links) means the next `G1 … S<cutting>` executes from the previous head position — **an unintended full-power chord across the workpiece**; a rejected mode line (M5/M4 S0) leaves the previous power mode active for the following group. The streamer then reaches `done` with `completed === total` — clean finish. `lastError` is write-only (grep: zero UI reads); `describeError` is exported and never called; the module's own contract (`response.ts:7-8`: "error:N — Line rejected; consume from buffer + surface") was implemented for buffer accounting only. The only trace is a red raw line in the passive LaserLog. The stale `error-codes.ts` header even claims a status-bar toast that does not exist.
- **Evidence:** `streamer.ts:138-144`: `const nextStatus = isAlarm ? 'cancelled' : …` — `laser-line-handler.ts:145-149`: `set({ lastError: cls.code }); advanceStream(…, 'error');` — `streamer.test.ts:71` pins "treats error like ok for buffer accounting (still consumes)".
- **LightBurn:** Surfaces controller errors prominently during a job; mainstream senders pause/halt rather than silently skipping motion lines mid-burn. No reference sender does "skip the line, keep cutting, report success."
- **Suggested fix:** On `error` ack while streaming: at minimum a persistent banner with `describeError(code)` and the rejected line; preferably feed-hold (`!`) or cancel, requiring operator acknowledgement. Render `lastError`/`lastWriteError` in the Laser panel. Mark the job "completed with N errors" instead of clean done. Keep consume-for-accounting.

#### H6. Jog and Home are usable while a job is streaming; their acks are misattributed to the stream head, corrupting the 120-byte RX accounting mid-burn

*(Merged: two independently-filed reports of the same defect; 4 adversarial verifications, all confirming.)*

- **Location:** `src/ui/state/laser-line-handler.ts:157` (`advanceStream`), `src/ui/laser/JobControls.tsx:119` (Home gate), `LaserWindow.tsx:75` (JogPad gate)
- **Status:** **confirmed** (structurally; the buffer-overflow endpoint is derived from GRBL's one-ack-per-line contract, not reproduced on hardware)
- **Description:** `advanceStream` attributes EVERY inbound ok/error to the streamer's head-of-flight line — no sender correlation exists. Frame/Start/Autofocus/Origin buttons are gated on `busy = disabled || streaming`, but the **Home button omits the streaming flag** and **JogPad is gated only on connection**. Mid-job `$H`/`$J=` are line commands GRBL answers with ok/error; that interleaved ack pops a stream line GRBL has NOT consumed, frees its bytes, and `step()` refills to 120 believed bytes while the un-acked line still occupies the real 128-byte buffer (8-byte margin) — plus the jog/home bytes themselves were never counted, so an immediate overflow is possible on the click itself. Consequences: corrupted/truncated G-code mid-burn (e.g. `G1 X100.0` arriving as `G1 X10` with the beam on), ack stream shifted one line for the rest of the job, early "done." The repo's own R-H2 comment names the failure: "accounting drift can push more bytes than the buffer holds → dropped commands → uncontrolled head motion."
- **Evidence:** `JobControls.tsx:119`: `disabled={props.disabled || !homingEnabled}` (siblings use `busy`, line 108) — `LaserWindow.tsx:75`: `<JogPad disabled={connection.kind !== 'connected'} />` — `streamer.ts:136-137`: `onAck` pops `inFlight[0]` unconditionally.
- **LightBurn:** Disables jog/home/move controls while a job is running; manual motion cannot be interleaved into the character-counted stream.
- **Suggested fix:** Gate JogPad and Home on streaming/paused exactly like the other controls. Longer term: serialize all line-oriented writes through one queue so every ack has an owner (CNCjs pattern), leaving only real-time single bytes (`?`, `!`, `~`, 0x18) out-of-band.

## 6. Beating LightBurn — roadmap

The maintainer's stated goal is a **better LightBurn, not a clone**. The research (parity + competitor landscape + LightBurn pain points) converges on a workflow-first plan in four stages. Each item names the structural advantage it exploits.

### Stage 0 — Earn the right to compete (trust foundation; weeks, not months)

You cannot market fidelity while the default configuration mirrors every photo engrave. Before any differentiation work:

1. **Fix the output-fidelity criticals/highs:** C2 (raster mirror), H3 (preview mirror), H1 (G92 fill/image block), H2 (hatch units), H12 (1024 cap), plus the import-fidelity trio H4/H8/H9. *Advantage exploited: the pure-core architecture makes every one of these a testable, pinnable fix.*
2. **Close the laser-off holes:** C3 (disconnect/unload stop), H5 (surface error:N), H6 (gate jog/home), H13 (kill native dialogs), and M22's keyboard stop. *Advantage: "not starting fires" is the user's #1 priority per PROJECT.md — and a marketable safety brand no hobby competitor takes seriously.*
3. **Make the suite able to see fidelity:** raster + fill perceptual fixtures, the centerline/skeleton metric, the H15 snapshot corpus extension, a 4-test Playwright smoke (M31), and run F.2.f/F.3 on hardware. *Advantage: Karpathy's-law discipline is already cultural here — finish the instrumentation.*
4. Resolve H11 (potrace provenance) — a legal overhang that gates any commercial launch.

### Stage 1 — Win the first ten minutes (the acquisition wedge)

LightBurn's funnel is download → trial activation → $99 → annual renewal friction. Ours can be a URL.

1. **Ship the PWA** (manifest + service-worker precache) so "fully offline, zero activation, no license server" is true on web — the exact ritual LightBurn documents pages of workarounds for. *Advantage: web delivery; LightBurn structurally cannot follow.*
2. **Own the orphaned platforms:** make Linux-Chromium a tested target (Playwright leg + udev/dialout connect hints); ChromeOS free. *Advantage: LightBurn publicly abandoned Linux in 2024; the cohort is vocal and shopping.*
3. **Named machine presets on top of $$ auto-config** (Falcon A1 Pro, Falcon 2, xTool D1 Pro, Sculpfun, Ortur) + a connect troubleshooter. First connect becomes near-zero-input — better than Find My Laser AND better than the vendor apps, which only detect their own brand. *Advantage: parse-settings.ts already reads the whole $$ dump; this is data + UI.*
