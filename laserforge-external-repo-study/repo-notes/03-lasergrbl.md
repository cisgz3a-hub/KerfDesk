## Purpose of This Repo in the LaserForge Study

This repo is being studied primarily for GRBL streaming, `M3`/`M4`/`M5` laser safety, raster/image engraving, beginner workflow, release packaging, practical Windows GRBL sender behavior, and G-code preview from emitted command state.

## 1. Executive Summary

LaserGRBL is a mature Windows C# GRBL sender focused specifically on laser cutters and engravers. It is not a broad architecture benchmark like Rayforge or MeerK40t; its value is practical sender behavior: buffered/synchronous streaming modes, GRBL status parsing, resume reconstruction, raster/image-to-G-code conversion, `M3`/`M4`/`M5` mode handling, preview from parsed G-code state, and simple installer packaging.

Build/test were not run because this environment does not have `msbuild`, `nuget`, `dotnet`, or `iscc`; this is recorded in `audit-artifacts/lasergrbl/build-test-status.txt`. ## 2. Why This Repo Matters to LaserForge

LaserGRBL matters because it provides concrete examples of:

- GRBL command parsing that recognizes G/M/S/X/Y/Z words anywhere in a line after helper parsing.
- Buffered GRBL streaming with pending-command accounting and serial-buffer byte budgeting.
- Separate realtime commands for feed hold, cycle start/resume, status query, door, jog cancel, and reset.
- Abort behavior that flushes queued commands and enqueues `M5` as a laser-off command.
- Resume reconstruction that replays modal state before continuing a partially executed job.
- Preview generated by running the same parsed command list through a state builder.
- Raster and vector image conversion paths that explicitly select `M3` or `M4` and force `M5` off.
- Installer packaging through an Inno Setup script.

These are external implementation lessons, not LaserForge defects. Each LaserForge action below requires a focused sector audit before any code change.

## 5. Architecture Map

LaserGRBL's practical sender architecture is:

```text
WinForms UI
  -> GrblCore
    -> GrblFile / GrblCommand / StateBuilder
    -> mQueue / mPending / mRetryQueue
    -> IComWrapper serial / Telnet / WebSocket wrappers
    -> GRBL status, error, alarm, buffer, and position parsing
  -> Preview UI reads GrblFile analyzed command state
  -> Raster/SVG/Potrace importers write GrblCommand lists
```

Important boundaries:

- `GrblCore` is a large central sender/control class, not a small pure domain module.
- `GrblFile` is both file/import model and preview/analyze source.
- `StateBuilder` is the parser-state lens used for analysis, preview, resume, and modal reconstruction.
- Transport uses byte-budgeted buffering in `GrblCore`, not a separate composable sender module.

LaserForge comparison targets:

- `src/controllers/grbl/GrblController.ts`

## 6. Safety Audit

LaserGRBL safety-relevant surfaces reviewed:

- `LaserGRBL/Core/GrblCore.cs`
- `LaserGRBL/GrblCommand.cs`
- `LaserGRBL/StateBuilder.cs`
- `LaserGRBL/GrblFile.cs`
- `LaserGRBL/RasterConverter/ConvertSizeAndOptionForm.cs`
- `LaserGRBL/SvgConverter/ConvertSizeAndOptionForm.cs`
- `LaserGRBL/CSV/*.csv`
- `README.md`

Observed safety patterns:

- `AbortProgram()` records a manual-abort issue, ends job timing, clears the queue, and enqueues `M5`.
- `FeedHold()` sends realtime `!`; `CycleStartResume()` sends realtime `~`; `GrblReset()` uses soft reset `0x18`.
- `RunProgramFromStart()` clears queues and clones loaded commands into the active queue.
- `ContinueProgramFromKnown()` analyzes earlier commands, enqueues `G90`, moves with `M5 G0 ...`, restores settled modals, then resumes remaining commands.
- `CanSend()` checks `mUsedBuffer + command.SerialData.Length <= BufferSize`.
- `ManageCommandResponse()` dequeues pending commands on `ok`/`error` and subtracts their byte length from `mUsedBuffer`.
- `ManageRealTimeStatus()` parses GRBL 1.1 status fields including `Bf`, `WPos`, `MPos`, `WCO`, `FS`, and overrides.
- `StatePositionBuilder.LaserBurning` models M4 dynamic-power behavior by treating `M4` on `G0` as non-burning and requiring `S > 0` when PWM is supported.
- Raster/SVG import UI offers `M3 - Constant Power` and `M4 - Dynamic Power`, but warns when M4 is selected and GRBL laser mode is not enabled.
- Generated raster/vector paths add `M5` at job/end boundaries.

Safety caveat:

- This study did not run LaserGRBL against real hardware or its emulator. These are patterns to compare against LaserForge, not proof that LaserGRBL's behavior is safe in all cases.

## 7. Streaming and Device-Control Lessons

### Useful Patterns

- Maintain a byte-counted pending queue for GRBL buffer budgeting.
- Treat realtime commands as separate from normal G-code stream buffering.
- Decode GRBL status reports into machine state, WCO, feed/spindle, and buffer signals.
- Record detected issues such as manual abort, unexpected reset/disconnect, stop responding, and machine alarm.
- Preserve last known WCO for resume.
- During resume, explicitly settle modal state before replaying the remaining job.

### Cautionary Patterns

- LaserGRBL's `GrblCore` is very large and mixes UI-era concerns with sender, queue, status, job, and configuration logic. LaserForge should learn behavior, not copy this shape.
- The test surface visible in the clone is weak compared with the risk surface: mostly Bezier/SVG flattening, not streaming/pause/abort/hardware transcript tests.
- The file model materializes imported G-code into `List<GrblCommand>`, which is practical for preview but not a model for LaserForge's large-job streaming direction.

## 8. Raster, Vector, and Preview Lessons

LaserGRBL's raster/vector pipeline shows useful practical details:

- Line-to-line raster conversion builds color segments and can optimize travel across zero-power segments.
- Raster processing supports horizontal, vertical, diagonal, dithering, vectorization, centerline, and Potrace workflows.
- `skipcmd` can be `G0` or `G1` depending on settings, which matters because GRBL laser mode treats `G0` as laser-off behavior.
- Preview derives from the same `GrblCommand` list used for output, then `StatePositionBuilder` updates modal state and burn state.
- Drawing ranges track moving range, drawing/burning range, and spindle/S range separately.

LaserForge comparison targets:

- Raster scanline/gap-bridge generation.
- Fill engraving and vector path ordering.
- Preview/output consistency.
- Whether preview uses the emitted plan/output state rather than UI guesses.
- Large-job streaming boundaries versus preview/export materialization.

## 11. What LaserForge Should Learn

### Copy Conceptually

- Keep GRBL parser helpers able to identify G/M/S words anywhere in the block after comment removal.
- Keep streaming byte-budget tests pinned around `ok`/`error` pending accounting.
- Keep realtime command separation explicit for hold, resume, status, reset, and jog cancel.
- Make resume reconstruct modal state and safe travel explicitly.
- Make preview consume the same parsed output/modal state as emitted G-code.
- Keep `M4` gated by actual laser mode, and treat M4 on rapid moves as non-burning in analysis.

### Do Not Copy

- Do not copy the large `GrblCore` god-class shape.
- Do not rely on thin test coverage for device-control behavior.
- Do not use materialized `List<GrblCommand>` as the only large-job representation in LaserForge.
- Do not copy GPLv3 code into LaserForge unless licensing compatibility is deliberately accepted.

### Improve LaserForge By

- Running focused sector audits for GRBL streaming, pause/resume/stop, preview/output consistency, and raster/vector generation.
- Checking whether LaserForge's command classifier/parser handles same-block embedded G/M commands as robustly as `GrblCommand.BuildHelper()`.
- Checking whether LaserForge's resume/start paths settle modal state and WCS/origin assumptions with explicit tests.
- Checking whether generated preview and burn-bounds analysis follow M4/G0/S semantics.
