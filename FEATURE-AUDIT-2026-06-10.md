## Live-pass findings the static auditors could not see

- **Centerline silently drops thick regions.** On a mixed image (square + stroke + triangle), only stroke-like features got centerlines; the 60×60 solid square produced nothing, with no warning in the commit toast. Correct-by-design for strokes, but a logo user loses content silently. Suggest a "N regions skipped (too thick for centerline)" note in the toast.
- **Trace preview is black-on-black.** Committed vectors render filled black over the dark source in the dialog preview; LightBurn draws trace results as colored strokes so fit is judgeable. Fade Image helps but the default view hides trace quality.
- **Double `M5` at the end of raster jobs** (emitted G-code, harmless but unclean).
- **Trace commit layer mode is preset-dependent** (Line Art → fill layer, Centerline → line layer). Looks deliberate and sensible, but PROJECT.md Phase E still says traces "flow through the existing Line pipeline" — needs a doc/ADR decision.
- Empty-workspace hint still reads "Drag an SVG here" though images are droppable (stale copy).
- Raster engrave G-code is structurally correct end to end: M4, per-pixel sweeps, travel `S0`, `$30=1000` scaling exact (30% → S300), `M5` close.

## Static auditor findings (full)

The sections below are the four auditors' findings verbatim (title, severity, location, fix suggestion). Items marked (fixed) were addressed on `fix/audit-hotfixes` this session.

### image-import — works-with-issues

The core image-import path is structurally sound and the two entry arms genuinely share one pipeline: both the drag-drop hook (src/ui/app/use-import-drag-drop.ts:73) and the toolbar/menu picker (src/ui/commands/CommandShell.tsx:195) call the same importImageFile (src/ui/commands/import-image-action.ts:12), which decodes, reads PNG-pHYs/JFIF density (96-DPI fallback), builds mm bounds from the natural size, stores original bytes as dataUrl plus a <=2048px luma, centers/fits to bed, auto-creates a #808080 image-mode layer, selects the object, and pushes one undo frame. However, sizing diverges from the LightBurn reference in two ways (96 instead of 254 DPI fallback; EXIF-resolution-only JPEGs ignored entirely), a crafted/corrupt PNG pHYs value of 1-19 px/m produces DPI 0 -> Infinity bounds -> NaN transform that silently bricks .lf2 saves and overwrites the autosave slot with an unreadable snapshot, multi-image drops stack every file pixel-exactly at bed center with no F-A3 stagger, and import leaves a pre-existing multi-selection's extras selected alongside the new image. The acceptance filters also drift between arms (drop strictly rejects webp/bmp/gif; the picker enforces nothing in code). Per Karpathy's law: none of the decode/placement/selection behavior is pinned end-to-end â€” importImageFile has no test of its own, the drop test mocks it, and jsdom cannot decode images â€” so everything visual (EXIF orientation, color, real placement) needs the live pass.

  - Where: `C:/Users/Asus/LaserForge-2.0/.claude/worktrees/audit-hotfixes/src/ui/common/image-density.ts:50`
  - pngDensity returns Math.round(ppuX * 0.0254), which is 0 for ppuX 1-19 even though the `ppuX > 0` guard passes. import-image-action.ts:26 forwards any non-null density, and rasterImportGeometry's `input.dpi ?? The object lands with Infinity bounds and NaN translation: invisible, unframeable, NaN-poisoning combinedBBox. Worse: serializeProject is bare JSON.stringify (src/io/project/serialize-project.ts:11), which writes Infinity/NaN as null; the load-side validator requires finite numbers (src/io/project/project-shape-validator.ts:345-346), so a manual Save As writes a .lf2 that can never be reopened, and the 30s autosave loop (src/ui/state/autosave.ts:115-121 â€” writes without validating) replaces any previous good autosave with a snapshot readAutosave will reject (autosave.ts:87-88), i.e. silent loss of the crash-recovery copy.   - Where: `C:/Users/Asus/LaserForge-2.0/.claude/worktrees/audit-hotfixes/src/ui/common/image-density.ts:69`

### trace — works-with-issues

The Image Trace feature is genuinely implemented end-to-end (Tools > Trace Image on a selected RasterImage -> ImportImageDialog with four binarized presets + LightBurn-vocabulary controls (Cutoff/Threshold band, Ignore Less Than, Smoothness, Optimize, Trace Transparency, Sketch Trace, Fade/Show Points/Boundary/Delete-after) -> shared worker-or-inline traceImageRegion -> potrace/centerline/imagetracer backends -> applyTraceToExisting overlay commit with a single undo entry), and preview and commit deliberately share the same trace function and the same 2048 px decode cap. WORKFLOW.md Phase E is confirmed a stub (WORKFLOW.md:667-672), and the controls that shipped implement ADR-030 B1-B4 while that ADR is still marked Proposed. Against the binding spec and LightBurn, three high-severity divergences stand: the kept source bitmap is silently excluded from G-code forever (contradicting ADR-026 section 5 and a scene-object comment that claims the role tag is display-only), committing a trace silently rewrites an existing same-color layer's mode (an existing black Line/cut layer becomes Fill), and committing a trace on any raster whose stored pixel grid exceeds the 2048 decode cap (Convert-to-Bitmap output over ~205 mm at default DPI, or pre-ADR-037 project files) produces a uniformly shrunk, misregistered overlay that the in-dialog preview does not show. Medium issues: Cutoff/Threshold/Ignore display values that are not the effective ones for Otsu presets, Smoothness/Optimize being dead controls in Centerline mode, and Escape during a commit closing the dialog without aborting the commit (Cancel is disabled while busy). The ADR-025 harness pins coverage IoU for the Line Art preset on five synthetic fixtures only; centerline fidelity, the other presets, and everything rendering/UX-level remain unpinned and need the live pass.
