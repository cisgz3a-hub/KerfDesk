### 0.5 Trust-boundary diagram (prose)

Data crossing trust boundaries in LaserForge:

1. **User filesystem ‚Üí renderer** ‚Äî `.laserforge.json` envelope load (`src/io/SceneSerializer.ts`, ~682 LOC). Envelope is versioned + migration-pipeline'd (T1-119). Adversary: hostile project file.
2. **User filesystem ‚Üí renderer (SVG / image)** ‚Äî `src/import/svg/SvgParser.ts` (606 LOC), `src/import/Dithering.ts` and raster import. Adversary: malformed SVG triggering DoS, unbounded coordinates, billion-laughs-style entity expansion.
3. **Renderer ‚Üí Electron main (IPC)** ‚Äî `electron/preload.ts` declares the bridge; serial/storage/wifi calls go through it. Adversary: compromised renderer (e.g. via malicious project file). The preload exposes a narrow surface; full enumeration is a Phase 4 task.
4. **Electron main ‚Üí USB/serial transport** ‚Äî `electron/serial.ts` writes to Falcon hardware. Trust boundary: bytes leave the program. Adversary: none (own hardware); but errors here cause physical damage.
5. **Electron main ‚Üí network (Falcon WiFi)** ‚Äî `electron/falcon-wifi/` (7 files, 1,066 LOC). HTTP + WebSocket bridge to a WiFi-connected Falcon. Adversaries: local-network attacker, off-by-one Falcon firmware versions, MITM if not TLS-pinned.
6. **Renderer ‚Üí controller (in-process)** ‚Äî `src/controllers/grbl/GrblController.ts` (2,733 LOC). Single-threaded message bus from MachineService/UI to wire-emitting code. Trust boundary in the IEC 62304 sense: this is the software safety boundary.
7. **Controller ‚Üí wire (gcode emission)** ‚Äî every byte that hits this boundary is irrevocable. Phase 5 (STPA) focuses here.
8. **Wire ‚Üí controller (status parsing)** ‚Äî `src/controllers/grbl/parse*.ts`. Adversary: garbled serial line (cable noise, firmware glitch). Audit checks fail-closed behaviour.
9. **Update channel ‚Üí installer** ‚Äî `electron-builder` config + `scripts/signing/`. Adversaries: typosquatted dependency, hijacked release. Out of scope for this audit (T3-4 / T2-99 / T2-100 are roadmap items).
10. **npm registry ‚Üí node_modules** ‚Äî build-time supply chain. Mitigated by `npm run license-check` (gated in `prepare`); `npm audit` covers known CVEs.

### F-001: ESLint baseline broken ‚Äî `usePositionTransform` false-positive in `SvgParser.ts`

- **Severity:** Low
- **Category:** Documentation / Maintainability (broken lint baseline)
- **CWE:** N/A

### F-002: `catch (e: any)` in WebSerialPort read loop

- **Severity:** Low
- **Category:** Type-safety
- **CWE:** CWE-755 (Improper Handling of Exceptional Conditions, weakly applicable)
- **ISO/IEC 25010:** Reliability (fault tolerance) ‚Äî partial; Maintainability
- **STPA / FMEA:** N/A (transient error, no laser-safety implication)
- **Risk score:** severity 2 √ó likelihood 2 √ó blast radius 1 = **4**
- **Location:** `src/communication/WebSerialPort.ts:532`
- **Evidence:**
  ```ts
  } catch (e: any) {
    if (this._readLoopActive) {
      this._errorCallback?.(new Error(`Read error: ${e.message}`));
    }
  }
  ```
- **Analysis:** The catch annotates `e` as `any`, then assumes `e.message` is a string. If the read loop throws something without a `.message` (e.g. a string, a number, an object with `message: () => string`), the constructed `new Error(\`Read error: ${e.message}\`)` produces `"Read error: undefined"` or `"Read error: function () { ... }"`. The transport-layer error is then propagated upward with degraded diagnostic value. Replace with `catch (e: unknown)` and the standard `e instanceof Error ? e.message : String(e)` pattern. Not exploitable; degrades support-bundle quality only.
- **Reproduction:**
  ```bash
  grep -n "catch (e: any)" src/communication/WebSerialPort.ts
  ```
- **Recommendation:** One-line fix:
  ```ts
  } catch (e: unknown) {
    if (this._readLoopActive) {
      this._errorCallback?.(new Error(`Read error: ${e instanceof Error ? e.message : String(e)}`));
    }
  }
  ```
- **Related findings:** F-001 (both improve type-safety / observability baseline).
