### 0.5 Trust-boundary diagram (prose)

Data crossing trust boundaries in LaserForge:

1. **User filesystem → renderer** — `.laserforge.json` envelope load (`src/io/SceneSerializer.ts`, ~682 LOC). Envelope is versioned + migration-pipeline'd (T1-119). Adversary: hostile project file.
2. **User filesystem → renderer (SVG / image)** — `src/import/svg/SvgParser.ts` (606 LOC), `src/import/Dithering.ts` and raster import. Adversary: malformed SVG triggering DoS, unbounded coordinates, billion-laughs-style entity expansion.
3. **Renderer → Electron main (IPC)** — `electron/preload.ts` declares the bridge; serial/storage/wifi calls go through it. Adversary: compromised renderer (e.g. via malicious project file). The preload exposes a narrow surface; full enumeration is a Phase 4 task.
4. **Electron main → USB/serial transport** — `electron/serial.ts` writes to Falcon hardware. Trust boundary: bytes leave the program. Adversary: none (own hardware); but errors here cause physical damage.
5. **Electron main → network (Falcon WiFi)** — `electron/falcon-wifi/` (7 files, 1,066 LOC). HTTP + WebSocket bridge to a WiFi-connected Falcon. Adversaries: local-network attacker, off-by-one Falcon firmware versions, MITM if not TLS-pinned.
6. **Renderer → controller (in-process)** — `src/controllers/grbl/GrblController.ts` (2,733 LOC). Single-threaded message bus from MachineService/UI to wire-emitting code. Trust boundary in the IEC 62304 sense: this is the software safety boundary.
7. **Controller → wire (gcode emission)** — every byte that hits this boundary is irrevocable. Phase 5 (STPA) focuses here.
8. **Wire → controller (status parsing)** — `src/controllers/grbl/parse*.ts`. Adversary: garbled serial line (cable noise, firmware glitch). Audit checks fail-closed behaviour.
9. **Update channel → installer** — `electron-builder` config + `scripts/signing/`. Adversaries: typosquatted dependency, hijacked release. Out of scope for this audit (T3-4 / T2-99 / T2-100 are roadmap items).
10. **npm registry → node_modules** — build-time supply chain. Mitigated by `npm run license-check` (gated in `prepare`); `npm audit` covers known CVEs.

### F-001: ESLint baseline broken — `usePositionTransform` false-positive in `SvgParser.ts`

- **Severity:** Low
- **Category:** Documentation / Maintainability (broken lint baseline)
- **CWE:** N/A
- **ISO/IEC 25010:** Maintainability (analysability, testability)
- **STPA / FMEA:** N/A
- **Risk score:** severity 2 × likelihood 1 × blast radius 1 = **2**
- **Location:** `src/import/svg/SvgParser.ts:369`
- **Evidence:**
  ```ts
  function resolveUseElement(useNode: Element, useTransform, ...): void {
    // ...
    const translatedTransform = multiplyMatrix(useTransform, usePositionTransform(useNode));
  ```
- **Analysis:** `usePositionTransform` is a pure SVG-parsing helper that computes a translation matrix from `x`/`y` attributes of a `<use>` element. Its name begins with `use` because it's about `<use>` elements, not because it's a React hook. The `react-hooks/rules-of-hooks` ESLint rule has no way to distinguish: any function whose name starts with `use` and is called from a non-component / non-hook context triggers the rule. Result: `npx eslint .` exits 1 on every run. **Safety impact: none.** **Process impact: real** — the lint suite has been red on this file; a real React-hooks violation could land elsewhere and be lost in the noise. The audit's static-analysis phase nominally requires lint to be clean as an entry criterion.
- **Reproduction:**
  ```bash
  npx eslint src/import/svg/SvgParser.ts
  ```
- **Recommendation:** Either (a) rename to `positionTransformForUseElement` (no behaviour change), or (b) add `// eslint-disable-next-line react-hooks/rules-of-hooks -- 'use' refers to the SVG element, not a React hook` immediately above the call site. ### F-002: `catch (e: any)` in WebSerialPort read loop

- **Severity:** Low
- **Category:** Type-safety
- **CWE:** CWE-755 (Improper Handling of Exceptional Conditions, weakly applicable)
- **ISO/IEC 25010:** Reliability (fault tolerance) — partial; Maintainability
- **STPA / FMEA:** N/A (transient error, no laser-safety implication)
- **Risk score:** severity 2 × likelihood 2 × blast radius 1 = **4**
- **Location:** `src/communication/WebSerialPort.ts:532`
- **Evidence:**
  ```ts
  } catch (e: any) {
    if (this._readLoopActive) {
      this._errorCallback?.(new Error(`Read error: ${e.message}`));
    }
  }
  ```
- **Analysis:** The catch annotates `e` as `any`, then assumes `e.message` is a string. If the read loop throws something without a `.message` (e.g. a string, a number, an object with `message: () => string`), the constructed `new Error(\`Read error: ${e.message}\`)` produces `"Read error: undefined"` or `"Read error: function () { ... }"`. The transport-layer error is then propagated upward with degraded diagnostic value. Replace with `catch (e: unknown)` and the standard `e instanceof Error ? e.message : String(e)` pattern. Not exploitable; degrades support-bundle quality only.
- **Reproduction:**
  ```bash
  grep -n "catch (e: any)" src/communication/WebSerialPort.ts
  ```
- **Recommendation:** One-line fix:
  ```ts
  } catch (e: unknown) {
    if (this._readLoopActive) {
      this._errorCallback?.(new Error(`Read error: ${e instanceof Error ? e.message : String(e)}`));
    }
  }
  ```
- **Related findings:** F-001 (both improve type-safety / observability baseline).

- **Analysis:** Both call sites emit diagnostic information that should appear in support bundles (e.g. the GRBL Machine size + homing-direction + laser-mode triplet is exactly the diagnostic tuple a user includes in a bug report).
