### F-001: `ControllerInterface.setStopOnError` signature doesn't surface the T1-116 token requirement

- **Severity:** Medium
- **Category:** Type-safety / API surface
- **Location:** `src/controllers/ControllerInterface.ts:399`, `src/controllers/grbl/GrblController.ts:520`
- **Evidence:**
  ```ts
  // ControllerInterface.ts:399
  setStopOnError?(value: boolean): void;
  ```
  ```ts
  // GrblController.ts:520
  setStopOnError(value: boolean, token?: UnsafeStopOnErrorOverrideToken): void {
    if (value === false && !isUnsafeStopOnErrorOverrideToken(token)) {
      throw new Error(...);
    }
    this._stopOnError = value;
  }
  ```
- **Analysis:** The implementation requires a non-empty-reason `UnsafeStopOnErrorOverrideToken` whenever `value === false`. The interface declares only `(value: boolean) => void`. A caller wired against the interface alone has no compile-time signal that the token is required — they could write `controller.setStopOnError(false)` and only learn at runtime (via a thrown `Error`) that the override gate exists. T1-116 added the runtime gate but didn't propagate the requirement into the interface signature.
- **Recommendation:** Update `ControllerInterface.setStopOnError` to `setStopOnError?(value: boolean, token?: UnsafeStopOnErrorOverrideToken): void` (after exporting the token type into a shared location, or accepting a structural shape). Lower priority because the runtime gate is the load-bearing defense, but the type signal would surface the contract at compile time.

### F-002: ConnectionGenerationGuard primitive shipped but never wired into transports

- **Severity:** High
- **Category:** Concurrency / Safety
- **Location:** `src/communication/ConnectionGenerationGuard.ts:19-23` (the "T2-34-followup" note), `src/communication/WebSerialPort.ts:52-54` (single-slot callbacks unchanged), `src/controllers/grbl/GrblController.ts:548+` (connect path registers callbacks without a generation tag)
- **Evidence:**
  ```ts
  // ConnectionGenerationGuard.ts:18-23
  // T2-34 ships the pure primitive (allocator + token + isStale check
  // + guard wrapper) so transport implementations and controllers can
  // adopt it without coordinating on a single connectionId source.
  // Wiring `WebSerialPort.onData/onError/onClose` + GrblController to
  // accept and forward `TransportCallbackContext` is filed as
  // T2-34-followup.
  ```
  Grep confirms no consumer:
  ```
  src/communication/TransportSubscription.ts        (mentions the type only)
  src/communication/ConnectionGenerationGuard.ts    (the module itself)
  ```
- **Analysis:** Audit 3B sections 9.4 + 8.4 + 3.8 identified the rapid disconnect/reconnect race: a stale microtask-queued read-loop event from a previous connection can fire against the new controller's callback because the single-slot callback was replaced. The fix primitive (`ConnectionGenerationAllocator`, `TransportCallbackContext`, `isStaleContext`, `withGenerationGuard`) shipped in T2-34. Production code paths (`WebSerialPort` + `GrblController`) have NOT adopted it. **This is exactly the "wired-into-product gate" violation CLAUDE.md T1-120 was added to prevent** — and it remained outstanding through 28 follow-up tickets without being retrofitted. Safety impact: a rapid reconnect could deliver a previous-connection RX line to the new controller, causing stale `<...>` status reports to influence safety-state classification at the moment a fresh controller is computing its safe-state-at-connect verdict.
- **Recommendation:** File `T2-34-followup` as a P0 retrofit ticket. Wire `WebSerialPort.onData/onError/onClose` to issue ctx-tagged callbacks; have `GrblController.connect` allocate a token and drop any RX events whose ctx generation doesn't match.

### F-006: `WebSerialPort` read loop catches `any` and accesses `e.message` unguarded

- **Severity:** Low
- **Category:** Robustness / Type-safety
- **Location:** `src/communication/WebSerialPort.ts:532-536`
- **Evidence:**
  ```ts
  } catch (e: any) {
    if (this._readLoopActive) {
      this._errorCallback?.(new Error(`Read error: ${e.message}`));
    }
  }
  ```
- **Analysis:** If `_reader.read()` rejects with something that isn't an `Error` (a `Symbol`, a thrown primitive, `null`), the `.message` access throws `TypeError: Cannot read properties of null`. The whole read loop unwinds without invoking the error callback. In practice browsers reject with `DOMException` (which is Error-like), so this rarely fires — but the guard exists everywhere else in the file (`e instanceof Error ? e.message : String(e)`) and was skipped here.
- **Recommendation:** Replace with `const message = e instanceof Error ? e.message : String(e);`. Trivial fix.

### F-007: `GrblController.disconnect` uses a hardcoded 50 ms delay before M5

- **Severity:** Low
- **Category:** Robustness
- **Location:** `src/controllers/grbl/GrblController.ts:790-792`
- **Evidence:**
  ```ts
  this._sendRealtime(REALTIME_FEED_HOLD);
  await new Promise(r => setTimeout(r, 50));
  this._port.write('M5 S0\n');
  ```
- **Analysis:** The 50 ms is the gap between sending feed-hold (realtime byte 0x21) and the subsequent M5. If firmware response is slower than 50 ms, the M5 might enter the planner while motion is still decelerating. The port closes shortly after, so the M5 may or may not have been transmitted before close. In practice this is "best effort" disconnect behavior with feed-hold as the load-bearing safety stop, but the magic 50 ms isn't documented or hardware-verified at the cited point.
- **Recommendation:** No urgent action. If we ever see "laser stayed on after disconnect" reports on a slow controller, the fix is to await a status report confirming hold or extend the delay.

### F-009: `_handleStatusReport` silently drops malformed status reports

- **Severity:** Low
- **Category:** Robustness
- **Location:** `src/controllers/grbl/GrblController.ts:1936-1937`
- **Evidence:**
  ```ts
  const parsed = parseGrblStatusReport(raw);
  if (parsed.stateWord === null) return;
  ```
- **Analysis:** If the status-report parser returns `stateWord === null` (malformed `<...>` envelope), the function returns without firing any listener and without logging anything. ### F-010: `MachineService._armAutoM5OnConnect` bypasses MachineCommandGateway

- **Severity:** Medium
- **Category:** Safety / Design / API surface
- **Location:** `src/app/MachineService.ts:1268`
- **Evidence:**
  ```ts
  // MachineService.ts:1265-1276 (inside _armAutoM5OnConnect callback)
  try {
    const verdict = getUnsafeAtConnect();
    if (verdict === null) {
      ctrl.sendCommand('M5 S0', 'internal');  // ← direct call, not via gateway
    }
  } catch {
    /* ignore: auto-M5 is defense-in-depth ... */
  }
  ```
  Confirmed by the existing source-pin test `tests/no-direct-sendcommand-outside-gateway.test.ts` which currently fails on master.
- **Analysis:** T3-90 added the post-connect auto-M5 defense-in-depth path but called `ctrl.sendCommand` directly instead of routing through `new MachineCommandGateway(...).sendCommand(...)`. The gateway is the load-bearing classifier for user commands and the future audit-log injection point; bypassing it means any future enforcement added at the gateway layer (rate-limiting, logging, etc.) won't see this M5. The path is `source: 'internal'` so the user-command-classification flow doesn't apply, but the no-bypass invariant test exists precisely to prevent the architecture from eroding. The test failure has been on master through 28+ subsequent ticket commits — nobody has filed a follow-up.
- **Recommendation:** Route through the gateway: replace with `new MachineCommandGateway(ctrl).sendCommand('M5 S0', 'internal')`. The gateway short-circuits source='internal' to a passthrough — no behavior change, and the source-pin test turns green.

### F-011: `MachineService.disconnect` swallows `operations.laserOff` errors without `notifyLaserSafetyOutcome`

- **Category:** Safety / Robustness

### F-014: `_armAutoM5OnConnect` listener leak on connect-failed-before-idle

- **Severity:** Low
- **Category:** Concurrency
- **Location:** `src/app/MachineService.ts:1257-1276`
- **Evidence:**
  ```ts
  const unsubscribe = ctrl.onStateChange((state) => {
    if (unsubscribed) return;
  ```

- **Analysis:** The listener auto-unsubscribes once status reaches 'idle'. Low absolute impact (one closure per failed connect) but accumulates over a session of repeated connect failures.
- **Recommendation:** Track the registered unsubscribe in a per-service field; clear it from `disconnect()` / `_handleTransportDisconnect()` if still armed.
