# Fill-pipeline change audit — 2026-06-03

> Adversarial audit of this session's fill-speed + fill-quality changes (1a overscan->rapid, 1b short-run skip, #2 continuous-sweep, interval 0.1mm). 7 review lenses -> per-finding code verification -> synthesis. Report-only.

## Verdict: fix-before-more-burns

This change set rewrites GRBL fill emission into a LightBurn-style continuous-sweep model (one G1 chain per scanline: ink spans burn at S{s}, interior holes cross at S0 with the diode dark), and threads the same sweep model through three consumers — the emitter, the planner ETA, and the preview scrubber — behind a single shared overscan helper. The core safety contract is intact: there is NO blocking burn hazard. The beam-state sequence across multi-hole sweeps is correct and exhaustively pinned (grbl-strategy.test.ts:215-260 asserts the exact `S300,S0,S300,S0,S300`), the geometry/coverage of the burn is preserved versus the prior emitter, and the one float value that gates byte output (the overscan short-run skip) is same-engine deterministic. The handful of zero-length / stationary-S edge cases that auditors found are real emit-contract violations but latent: they require a degenerate or touching span that the live fillHatching producer does not emit (0 degenerate runs out of 200 even at 0.05mm spacing), and even if one reached the emitter, GRBL laser mode ($32=1, default-enabled on the Falcon profile) holds the diode off on a zero-distance move. So the safety dimension is sound. The real, repeated problem is TEST COVERAGE on net-new code that IS wired into production. Three independent major findings converge: the planner's multi-span ETA pricing (appendFillGroupBlocks), the preview's interior-gap branch (appendFillSweepSteps), and the determinism property/fuzz harness all leave the new sweep path unexercised — `git diff` of both planner.test.ts and toolpath.test.ts is empty, and the determinism fuzz only ever feeds `kind: 'cut'` groups so the new float-projection comparator, the 1e-6 grouping epsilon, and the S0 emitter never see a fuzzed fill. The project's own contracts make this a gate, not a nicety: CLAUDE.md's wired-into-product rule (T1-120) requires an integration test that exercises the live path, PROJECT.md:348 lists the 100-seed determinism fuzz as a Phase-A acceptance gate, and the safety bar demands tests cover the failure mode on any g-code emission. The recommendation is to add the missing regression and fuzz coverage (a few focused tests, no production code change required) before the next hardware burn, then ship. The end-state runtime behavior is correct; what is missing is the net that keeps it correct under the next edit.

### Determinism property/fuzz harness never exercises fill groups — the new sort/projection/epsilon-grouping code has zero cross-run coverage despite being a stated Phase-A acceptance gate
- **Location:** `src/core/output/grbl-strategy.property.test.ts:49-50 (arbGroup hard-codes kind: fc.constant('cut')), :69-79 (determinism fuzz)`
- **Fix:** Extend arbGroup to also generate fill groups — e.g. add a fill arbitrary that produces multi-span segments on shared scanlines at randomized angles, and feed it through the existing emit(job)===emit(job) determinism property at numRuns:100. This is the single highest-value gap: it is the exact net PROJECT.md:348 names as a Phase-A gate, and it is the only thing that would catch a future regression in buildSweep's float-projection comparator (fill-sweeps.ts:91, proj = p.x*ux+p.y*uy) or the COLLINEAR_EPS_MM=1e-6 grouping. The code is deterministic today, so this is a regression guard, not a live-bug fix.

### Planner multi-span fill-ETA path (appendFillGroupBlocks) has zero test coverage and is live in production via estimateWithPlanner

- **Fix:** Add a planner.test.ts case with a FillGroup whose segments form a multi-span sweep (e.g. spans 0-5, 8-12, 15-20 on one scanline) and assert the ETA breakdown: one cut block priced burnStart..burnEnd over the whole envelope (holes priced at cut, per ADR-034's documented bias) plus the runway at travel velocity, and that the total matches gap-at-feed motion. ETA is advisory (no burn actuation), which is why this is should-fix not blocking, but it is net-new code on a live user-facing path that CLAUDE.md's wired-into-product gate requires be exercised.

### Toolpath preview multi-hole sweep (appendFillSweepSteps) has zero test coverage of its interior-gap branch

### No regression test covers touching/tangent/zero-length spans on the continuous-sweep emit path, despite the heightened test bar on g-code emission
- **Location:** `src/core/job/fill-sweeps.test.ts:11-65; src/core/output/grbl-strategy.test.ts:130-261`
- **Fix:** Add two emit-level assertions: (1) feed a sweep with two touching full-length spans (e.g. 0->5 and 5->10 on y=5) and assert NO zero-length G1 is emitted (currently it emits a redundant `G1 X5 Y5 S0`); (2) feed a mixed sweep containing a degenerate span (start===end) and assert NO stationary positive-S G1 is emitted. The existing negative assertion `expect(out).not.toMatch(/^G1[^\n]* S0$/m)` lives only in the single-span overscan test (grbl-strategy.test.ts:173) and never reaches the multi-span gap-crossing path where the zero-length variants arise. These tests also document the failure modes the emitter guard (below) should close.

### Emitter has no defensive zero-length / coincident-span guard; the 'positive S only on a MOVING G1' invariant relies entirely on the upstream fillHatching filter two modules away, and findLaserOnTravelIssues does not flag stationary G1s
- **Location:** `src/core/output/grbl-strategy.ts:138-151 (sweepSpanLines); src/core/invariants/predicates.ts:59-86 (findLaserOnTravelIssues gates only on /^G0\b/)`
- **Fix:** Add a defense-in-depth guard in sweepSpanLines: skip the ink-burn line when span.start===span.end (or |end-start| < epsilon) and skip the gap line when next.start===span.end. Optionally extend findLaserOnTravelIssues to also flag a positive-S G1 whose target equals the current head position (a stationary beam-on move), so the invariant net covers G1 and not just G0. Confirmed today this path is unreachable from production (fillHatching's SCANLINE_EPS filter drops degenerate runs before they reach the emitter), so this is hardening a latent trap on a safety-bar path, not patching a live exploit — hence should-fix rather than blocking.

## Strengths (6)

- The safety-critical beam-state sequence is exhaustively pinned at the emit level: grbl-strategy.test.ts:215-260 asserts the exact S sequence S300,S0,S300,S0,S300 for a multi-hole scanline. A missed S0 across a hole is the one thing that would fire the beam over empty space, and that exact failure mode is the one the suite locks down hardest — the right place to spend test budget on a laser.
- The continuous-sweep rewrite is a genuine, well-executed step toward the project's hard problem (fill fidelity vs LightBurn): it collapses thousands of stop-start runs into a few hundred continuous sweeps while provably preserving the positive-S span SET and the snake ordering (buildSweep orients every span along the first run then sorts by projection, so reverse rows naturally run right-to-left). Burned coverage is identical to before; only laser-off travel changed.
- The overscan runway decision is correctly centralized: a single effectiveOverscanMm helper is the one source of truth, and all three consumers (emitter grbl-strategy.ts:121, planner planner.ts:129, preview toolpath.ts:76) call it with byte-identical envelope endpoints. This is exactly the right structure to keep emit, ETA, and preview from drifting — the remaining gap is only that no test pins it.
- Determinism is preserved in the end state: the grouping epsilon (COLLINEAR_EPS_MM=1e-6) sits ~50,000x below the minimum hatch spacing (0.05mm), so two parallel scanlines can never accidentally merge, and the coordinate formatter (toFixed(3)) quantizes all emitted positions. The one unrounded float gate (the overscan skip) is same-engine reproducible, satisfying PROJECT.md invariant #5 as written.
- The module-boundary and pipeline discipline holds: the new code lives where it should (fill-sweeps.ts in job/, emission in output/), the fill path is genuinely wired into production (compile-job.ts feeds FillGroup.segments from fillHatching; estimate-duration.ts:41 routes through estimateWithPlanner), and the design decisions are recorded in ADRs 033/034 with explicit, honest Consequences (including the documented cut/travel ETA-breakdown bias and a standing 'hardware verification pending' note).
- The edge-case defects auditors found are correctly bounded and self-aware: every one was downgraded to minor/nit on verified reasoning (GRBL laser mode suppresses the beam on zero-distance moves; the degenerate path is unreachable from the live producer; the redundant gap line carries S0 so beam state is always correct). The change set never produces an unsafe beam state on any reachable input.
