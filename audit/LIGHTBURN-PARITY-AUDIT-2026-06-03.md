## Executive summary

LaserForge-2.0 has a sound pipeline (Scene→Job→Plan→Output→Device), color-as-layer parity, working Line/Fill/Image modes, GRBL streaming, preflight gating, text via opentype.js, and trace via potrace/imagetracerjs. But three structural gaps block real LightBurn parity, and one safety gap should be closed first.

THE FOUR BIGGEST PARITY GAPS (in dependency order):

1. NO SHAPE CREATION / EDITING (Tier 1, P0). You cannot draw a rectangle, ellipse, line, or polygon, cannot edit nodes, cannot boolean/weld/offset, cannot group, and cannot snap. Geometry enters ONLY via SVG/image/text import. For a "LightBurn-style CAM app" this is the single largest divergence: a user who can't draw a box can't do the most common laser task. Several of these (boolean, offset, node-edit) are explicitly out-of-scope per PROJECT.md and must NOT be built without a PROJECT.md revision + ADR — but parametric primitive creation and grouping are NOT on the out-of-scope list and are the highest-value in-scope wins. This requires a new SceneObject variant strategy and a real tool/interaction layer.

2. CUT-ORDER CORRECTNESS — inside-first ordering and layer reordering (Tier 0/1, P0/P1). optimize-paths.ts is pure nearest-neighbor with NO containment analysis, so an outer contour can be cut before its interior features, freeing the part before cutouts finish — a quality AND safety defect. Layers are also fixed at import order with no Move Up/Down. Both are squarely in scope (§8.6 backlog item 7) and touch the pure core cleanly. Inside-first is the highest-leverage single fix for output fidelity that the test suite does not measure.

3. THE CUT SETTINGS MODEL IS TOO THIN (Tier 1, P1). Single power (no Min Power), no M3/M4 vector toggle, only 3 dither modes (vs LightBurn's ~10, Jarvis being the photo default), no per-image tonal adjust/scan-angle/threshold/overscan exposure, no sub-layers, no air-assist (M7/M8). These are the per-operation knobs that decide whether a burn "looks right." Min Power + M4 and the Jarvis/Stucki/Atkinson/Ordered dither set are the parity-critical subset; many others are minor.

4. NO START-FROM / JOB-ORIGIN UI (Tier 1, P1). The 9-anchor JobOriginAnchor and applyJobOrigin math ALREADY EXIST in core but are hardcoded to front-left and never surfaced; JobStartMode lacks 'current-position'. This is a "wire the latent capability + add the UI" job, not greenfield — high value for low core risk.

RECOMMENDED SEQUENCE: Start with Tier 0 (inside-first ordering + the laser-off-on-travel preflight gate) because both are pure-core, test-friendly, fidelity/safety wins that need no schema change and no new UI surface. Then Tier 1 foundations: layer reordering + recolor/assign (small store actions), shape-creation foundation (new ParametricShape SceneObject variant + a Tool/interaction layer), Start-From/9-dot UI (wire existing core), and the dither-set + Min/Max-power redesign (which the project already scoped as §8.6 items 3 and 5). Tier 2 covers important workflow (Optimization Settings surface, numeric W/H + align/distribute, snapping, Convert-to-Bitmap completion, raster tonal adjust). Tier 3 is advanced/niche (offset-fill, cross-hatch, perforation, tabs, arrays — several gated behind PROJECT.md revisions).

HARD CONSTRAINT: PROJECT.md explicitly lists Boolean ops, node editing, offset/kerf, tabs/bridges, lead-in/out, arrays, variable text, system fonts, camera, rotary, and DXF/PDF import as OUT OF SCOPE. The audit guardrails forbid recommending them as silent work. Every such item below is flagged "REQUIRES PROJECT.md revision + ADR (stop-and-ask)" and placed in Tier 3 so the in-scope parity wins are never blocked behind a scope fight. The fidelity problem the test suite does NOT measure (does the trace/fill/engrave look right) is addressed primarily by inside-first ordering, the expanded dither set with Jarvis default, Min/Max power, and raster tonal adjustment — these should be hardware-verified on the Falcon, not just unit-tested.

### Tier 0 — Foundations & safety — pure-core fidelity/safety wins needing no schema change, no new UI, and directly testable

#### Inside-first (containment) cut ordering  `P0` · effort L
- **LightBurn:** LightBurn's cut planner orders nested inner contours before the outer boundary that contains them ('Cut inner shapes first', ON by default), so a cut-out part is not released from the stock before its interior features finish. A core safety/quality optimization.
- **Current:** optimize-paths.ts performs pure per-layer nearest-neighbor with NO containment analysis (pickBestNext chooses purely by squared Euclidean distance; segmentEntries handles only open-vs-closed). An outer contour can be cut before interior features, dropping/shifting the part mid-job. Confirmed by reading optimize-paths.ts in full and LIGHTBURN-STUDY.md §8.6 item 7.
- **Approach:** Add a pure containment-ordering stage in src/core/job/ that runs BEFORE nearest-neighbor within each CutGroup/FillGroup. Build a point-in-polygon (even-odd ray cast) helper (reuse the scanline logic already present in src/__fixtures__/perceptual/rasterize.ts as a reference, but implement a dedicated pure helper in core/job to respect boundaries). Compute a containment forest over closed segments (a segment B is inside A if a representative vertex of B is inside A's polygon AND B's bbox ⊂ A's bbox), assign each closed contour a depth, and emit deepest-first; feed the depth-bucketed order as the seed/constraint to nearestNeighborOrder so travel optimization still runs WITHIN a containment tier but never reorders an outer contour ahead of its children. Open polylines (no containment) keep current behavior. Keep deterministic tie-break by original index. Add G-code snapshot tests with nested-shape fixtures (circle-in-square, 3-level nesting) asserting inner segments precede outer.
- **Modules:** src/core/job/optimize-paths.ts, src/core/job/ (new containment-order.ts), src/core/job/job.ts (types if needed), src/io/gcode/emit-gcode.ts (consumes optimized job — no change expected)

## Completeness critic

This is an unusually rigorous, well-verified plan. I confirmed its four headline diagnoses against the live tree: optimize-paths.ts is pure nearest-neighbor with zero containment analysis (verified, full read); findLaserOnTravelIssues/expectedS/collectG1SValues are referenced ONLY in tests + the invariants re-export, never in runPreflight/emitGcode (verified via grep — 5 files, all test/index); the 13-mode dither kernel set lives only in dither-trace.ts (RGBA→trace) while dither.ts (S-values→engrave) has exactly 3 modes (verified); and the 9-anchor JobOriginAnchor math exists in job-origin.ts but every caller hardcodes USER_ORIGIN_JOB_PLACEMENT/front-left (verified). The tier ordering (pure-core safety/fidelity first, then structural parity, then niche) is sound, and the PROJECT.md out-of-scope guardrails are handled correctly.

However, the plan has a SAFETY BLIND SPOT that is serious given the task's explicit framing ("firing during pause"). Its only safety item (Tier-0 #2) hardens the STATIC g-code laser-off-on-travel invariant — a defense-in-depth wiring win, but it does NOT touch the far more dangerous RUNTIME failure modes that the project's own PROJECT.md non-negotiables call out (#9 "E-stop reachable always"; the pause/feed-hold path). I read laser-store.ts pauseJob: it writes RT_HOLD ('!') and nothing else. On GRBL, feed-hold decelerates but the spindle/laser PWM is only cut by firmware when $32=1 laser mode is active AND the deceleration completes; during the decel ramp the beam stays on, dwelling over the workpiece — a real scorch/fire risk. There is no M5-on-pause, no verification that pause actually extinguishes the beam, and no test covering "pause mid-G1-burn." This is the single most important MISSING item and it is squarely in the plan's stated "hard problem + safety" remit. Two other safety/correctness gaps are also absent (see safetyOrFidelityRisks). On fidelity, the plan correctly prioritizes inside-first + dither + Min/M4, but UNDER-prioritizes "remove overlapping lines" (double-burn on shared edges is a visible quality defect it leaves at the bottom) and entirely omits any perceptual/visual fidelity regression test harness, despite the codebase already having a perceptual rasterize fixture (src/__fixtures__/perceptual/rasterize.ts) that the inside-first item even proposes to borrow from. The plan repeatedly says fidelity work "should be hardware-verified, not just unit-tested" but never proposes the one thing that could catch fidelity regressions in CI: a render-and-compare harness. Net: adopt the plan's sequence, but insert a pause-safety ticket into Tier 0 ahead of (or alongside) the static-invariant gate, and add a perceptual-diff test harness as a Tier-0/1 foundation.

**Safety / fidelity risks flagged:**
- RUNTIME PAUSE: pauseJob sends only RT_HOLD ('!') with no M5 and no verification the beam is extinguished; GRBL feed-hold can leave the laser lit during deceleration / when $32 is misconfigured, dwelling over the work. Unaddressed by the plan. (Verified: laser-store.ts lines 409-413; commands.ts RT_HOLD='!'.)
- STOP/E-STOP: stopJob relies on RT_SOFT_RESET (Ctrl-X). The plan never proposes verifying PROJECT.md non-negotiable #9 ('Stop reachable from any window state; no modal can block it') — and several plan items add NEW modals/dialogs (Optimization Settings modal, 9-dot selector, shape tools). Each new modal is a chance to violate #9; the plan adds no guard or test that Stop stays reachable while they are open.
- FIDELITY REGRESSION DETECTION: the plan changes the default raster dither to Jarvis and adds a tonal pre-pass, fill cross-hatch, and inside-first reorder — all of which alter rendered output — yet proposes NO automated visual-diff coverage, only deterministic byte snapshots (which it admits will break on the Jarvis default change) plus manual hardware checks. The existing perceptual rasterize fixture is referenced but never elevated into a reusable fidelity harness. Fidelity changes are therefore unfalsifiable in CI.
- RASTER ETA UNDERCOUNT: planner.ts skips raster groups (`continue`), so the job-time estimate the plan surfaces in Preview/Optimization is silently wrong (too low) for the longest jobs. Not flagged in the plan; risks the operator walking away from a job they think is short.
- SVG IMPORT GEOMETRY: transforms/<use>/<symbol> are not composed and unit suffixes are dropped on import, so a transformed/nested SVG burns at the wrong size/position TODAY. A wrong-scale cut on real material is a fidelity AND material-waste/safety-adjacent defect; the plan ranks the fix Tier 3.
- Min Power vs device $31: introducing a per-layer minPower without reconciling it against the firmware $31 floor (which controller-readiness already warns about) could produce hotter-than-expected low-power burns — a fidelity/scorch risk the Min Power ticket does not address.

**Items the plan missed or under-weighted:**
- **Verify and harden pause/feed-hold actually extinguishes the beam (runtime 'firing during pause')** (P0 (Tier 0)) — THE critical omission. laser-store.ts pauseJob (lines 409-413) sends only RT_HOLD '!'. GRBL feed-hold decelerates motion but does NOT guarantee the laser is off during the decel ramp, and on $32=0 (or mid-G1) the diode can dwell lit over a stationary/slowing head — a burn-through/fire risk. The plan's only safety item hardens STATIC g-code (every G0 has S0), which says nothing about the runtime hold case. PROJECT.md non-negotiable #9 ('E-stop reachable always; no modal can block it') and the CLAUDE.md safety-path bar ('leave the laser firing during pause' is named as the cost of being wrong) both demand this. Needs: confirm GRBL real-time behavior on the target Falcon, consider emitting/relying on laser-mode auto-off, add a test that exercises pauseJob mid-burn, and a 'Hardware verification needed' note. This is a coupled triple touching a safety path (laser-store/streamer) and must precede or accompany the cosmetic static-gate item.
- **Perceptual render-and-compare test harness for trace/fill/raster fidelity (CI-enforceable)** (P1 (Tier 0/1 foundation)) — The plan's stated hard problem is 'does the burn LOOK right,' and it repeatedly defers this to manual hardware verification for inside-first, dither, Min/M4, and tonal-adjust. But the repo ALREADY has src/__fixtures__/perceptual/rasterize.ts (the inside-first item proposes borrowing its scanline logic). A pure rasterize-the-emitted-toolpath-and-diff-against-golden harness would let every fidelity ticket (dither default change to Jarvis, fill hatching, inside-first reorder, tonal pre-pass) prove it didn't regress the rendered output deterministically — exactly the gap the deterministic-G-code snapshot tests cannot cover (they assert byte-equality, not visual correctness, and the plan even flags that changing the Jarvis default 'changes byte output, coordinate with snapshot tests'). Without this, every fidelity change is unfalsifiable until hardware. Should be a Tier-0/1 foundation so subsequent fidelity tickets plug into it.
- **Remove overlapping / coincident lines (shared-edge double-burn)** (P1 (Tier 1, alongside inside-first/Optimization Settings)) — The verified-gaps list flags this as a real gap (optimize-paths only reorders, never dedups; compile-job appends every matching path's polylines verbatim). It is a VISIBLE output-fidelity defect — adjacent shapes sharing an edge get cut twice, charring the seam and wasting time — which is precisely the 'does it look right' problem the task foregrounds. The plan buries it implicitly under the Tier-2 Optimization Settings surface ('any later planner options') without giving it its own ticket. It belongs called out as a fidelity item near inside-first, not as an unscheduled 'future toggle.'
- **Raster job-time estimate (planner skips raster groups entirely)** (P2 (Tier 2, with the estimate/preview surface work)) — Not in the plan at all. I confirmed via the plan's own verified-gaps and the code path that planner.ts does `if (group.kind === 'raster') continue;` and estimate-duration.ts has no separate raster path — so image-engrave time is uncounted and the ETA badge silently under-reports for any photo/raster job (often the longest jobs). This is a correctness/trust gap in a surface the plan touches (it adds the Optimization toggle to live-job-estimate). Worth a small ticket; at minimum it should be noted as a known-wrong estimate so the Optimization Settings/preview work doesn't inherit a silently-broken number.
- **Surface controller-readiness warnings ($31 min-power nonzero) and the existing safety gates in the new Min Power / M4 UI** (P1 (fold into the Min Power/M4 ticket)) — The Min Power + M4 ticket (Tier 1) revises ADR-020 and adds a per-layer minPower, but controller-readiness.ts already emits a 'min-power-nonzero' WARNING when device $31>0 ('Low nonzero power values may burn hotter than expected'). Introducing a per-layer Min Power that maps to low S values interacts directly with that firmware $31 floor — the plan does not mention reconciling the new per-layer minPower against the $31 device clamp, which could double-apply or surprise the operator. The ticket should explicitly cover that interaction and keep the warning meaningful.

**Mis-prioritized:**
- 'Remove overlapping lines' is effectively unscheduled (folded vaguely into Tier-2 Optimization Settings 'future toggles') despite being a verified, visible output-fidelity defect. It should be a named Tier-1 fidelity ticket adjacent to inside-first ordering.
- SVG import fidelity (transforms/<use>/<symbol>/units) is placed in Tier 3 with priority field 'P2' — an internal contradiction the plan half-acknowledges. Transformed/nested SVGs currently import at the WRONG position/scale (verified: parse-svg.ts never reads transform, parseViewBox drops unit suffixes). Since SVG is the ONLY vector import path and geometry can ONLY enter via import until shape-creation lands, silently-wrong import geometry is a higher-frequency correctness defect than its Tier-3 placement implies. The regression risk is real, but it deserves Tier-2 scheduling, not Tier-3 burial.
- The Tier-0 safety item is mis-scoped as THE safety win when it only addresses static g-code. Given the task's explicit 'firing during pause' framing and PROJECT.md non-negotiable #9, runtime pause-safety (see missing items) is the higher-priority Tier-0 safety work; the static-invariant gate is genuinely valuable defense-in-depth but is lower-severity than the plan's prominence suggests.
- Air-assist per-layer (M7/M8) is Tier 2, but it is a safety-RELEVANT emission change (M7/M8 interacts with M3/M4/M5 ordering, and air assist is a fire-mitigation control on many materials). The plan flags it 'SAFETY-adjacent' yet ranks it below several pure-convenience editor features. Its emission-ordering test burden argues for treating it with the same heightened bar as the other g-code-emission tickets.
