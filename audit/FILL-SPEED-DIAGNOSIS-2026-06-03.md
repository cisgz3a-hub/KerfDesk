# LaserForge 2.0 — Fill Burn Speed Diagnosis (2h vs LightBurn 5min)

> Generated 2026-06-03 by a 17-agent workflow (5 code-diagnosis + 6 LightBurn/GRBL research + 1 docs -> 3 bottleneck-quantify -> architect + safety critic). Report-only.
> Repo: `C:\Users\Asus\LaserForge-2.0` @ `codex/main-working`. Code claims verified against source; LightBurn/GRBL behavior from cited web sources.

## Root-cause summary

The ~24x gap (≈2h vs LightBurn's ≈5min) on a traced-image Fill is NOT one bug and is NOT the classic GRBL streaming story. Verified against the live tree at C:/Users/Asus/LaserForge-2.0, in priority order:

CAUSE 1 (DOMINANT, ~50-70% of the 2h) — Per-run overscan traversed at CUTTING feed on every interior hatch run. grbl-strategy.ts emitFillGroup (lines 92-99) expands EACH hatch run via expandFillHatchWithOverscan (fill-overscan.ts:23-31, default 5mm/side = 10mm runway, layer.ts:49) and emits the lead-in as `G1 ... F{feed} S0` (line 96) and the lead-out as `G1 ... S0` inheriting modal F (line 98) — i.e. 10mm of laser-OFF travel per run at the cutting feed (default 1500 mm/min = 25 mm/s, layer.ts:48), NOT a rapid. planner.ts:120-123 confirms the estimator itself prices both lead moves at cutV. At 25 mm/s, 10mm = ~0.4-0.5s of pure overscan PER RUN, independent of burn length. A traced image fragments each 0.2mm scanline into many short runs, so on the short runs the 10mm runway dwarfs the actual ink.

CAUSE 2 (~20-35%) — Forced full decel-to-zero + accel-from-zero at BOTH ends of every run. planner.ts:239 `if (prev.kind !== next.kind) return 0` makes every travel→cut and cut→travel junction a full stop; appendFillGroupBlocks (planner.ts:118-124) sandwiches each burn between travel blocks, so the head stops entering AND leaving every burn. With accel=500 mm/s² (device-profile.ts:87) a sub-few-mm run is a pure accel/decel triangle that never reaches feed. This happens in SOFTWARE structure regardless of the controller's $32.

CAUSE 3 (THE MULTIPLIER) — Run-count explosion with no merging and fill excluded from path optimization. fill-hatching.ts:230-247 emits one isolated 2-point polyline per even-odd interior run per scanline; compile-job maps each 1:1 to a CutSegment; optimize-paths.ts:54-57 returns early unless kind==='cut', so FillGroups are never reordered or merged. A 100mm-tall trace = ~500 scanlines × 5-40 runs/row = ~1,250 (modest) to 10,000-20,000 (halftone) isolated runs, each paying Causes 1+2. This is what scales the per-run tax to ~1-2h. Perf fixture models 35,000 contours (fill-hatching.perf.test.ts).

CAUSE 4 (host-side back-pressure, ~1.5-4x but ONLY at high feed) — Per-ack main-thread render + forced reflow throttles ack-consume-and-resend. The streamer is correctly character-counting (streamer.ts:106-115, 120-byte window) — the classic 'send-and-wait per line' 24x mechanism is REFUTED. But web-serial.ts:181 fans every inbound line synchronously to subscribers on the renderer main thread; handleLine (laser-line-handler.ts:111) does set({log}) on every line incl. every 'ok', then advanceStream (line 168) does a SECOND set({streamer}) before the next bytes go out (line 169). LaserLog (LaserLog.tsx:13,27,46-51) subscribes s.log, filters+maps up to 200 rows unmemoized/unvirtualized, and reflows via scrollHeight/scrollTop/clientHeight on every log change (lines 20-25). No worker or Electron-side streaming offload exists (workers are trace-only). So ~2 saturated 200-row renders+reflow per ack serialize with sending. At the DEFAULT 1500 mm/min the required host rate is only ~7-10 lines/sec and even a throttled host meets it (contribution ≈ 0); this only binds when feed is high enough (≈90-250 lines/sec) to exceed the per-ack ceiling.

SECONDARY/REFUTED: Fill uses M3 constant power (preamble M3 S0; emitFillGroup constant S); M4 is raster-only (emit-raster.ts:77, ADR-020). M3-vs-M4 is ~0 on time (quality only). The host never emits $32=1 — laserModeEnabled is parsed from the controller's $$ and only drives a preflight WARNING (controller-readiness.ts:63-67); if the user's controller has $32=0 it would add a real per-S-change stop multiplier on top, but Causes 1-3 already reproduce 24x with $32=1. No mm/min↔mm/s feed bug (grbl-strategy.ts:85 emits F verbatim). Traced image is vector-hatched, NOT per-pixel rasterized in Fill mode (compile-job routes traced-image→fill contours→fillHatching). The 120-byte RX window gives only ~4-5 short lines in flight — real but minor on its own.

## Where the ~2 hours goes

For a representative dense/halftone traced-image Fill (100mm tall, 0.2mm spacing = ~500 scanlines, ~15-20 runs/row ≈ 7,500-10,000 runs) at the verified default 1500 mm/min / accel 500 mm/s² / 5mm overscan:

- ~50-70% — Overscan-at-cutting-feed: ~0.4-0.5s of laser-OFF runway per run × 7,500-10,000 runs ≈ 56-83 min. This is the single largest slice. On the short runs a trace produces, the fixed 10mm runway exceeds the actual burn length several-fold.
- ~20-35% — Per-run accel/decel triangle (full stop both ends): the head never reaches commanded feed on short runs; fixed start/stop overhead × thousands of runs ≈ 20-40 min.
- ~5-10% — Inter-run G0 rapids (one per run; fast at 6000 mm/min but each is a fresh accel-from-rest) ≈ a few minutes.
- <5% — Actual productive burning (the ink itself). The overwhelming majority of wall-clock is non-burning overhead.
- Host-side back-pressure (Cause 4) — at default feed contributes ≈ 0 to this breakdown; it would only appear (and could add a 1.5-4x multiplier ON TOP) if the user raised feed toward LightBurn's settings (e.g. 6000 mm/min) where the required host line-rate (~90-250/sec) exceeds the per-ack render+reflow ceiling (~60-160/sec).

Net: motion/emit structure (Causes 1+2+3) accounts for essentially the entire ~2h at default feed. LightBurn burns the same artwork as a continuous raster: overscan paid ONCE per row (~500 rows × 10mm ≈ 5m of overscan ≈ 2-3 min total) with no full stop between features within a row — which is the ≈5min reference.

## Expected result if fixed

With P0 #1 + P0 #2 (and the cheap P1 #4) done, the Fill emit structure converges on LightBurn's continuous-raster shape: overscan paid roughly once per row instead of per run, no full stop between features within a row, and ~500 long sweeps instead of ~10,000 short runs. Expected reduction is multiplicative — roughly (3-5x from #1) compounded with the structural collapse from #2 — bringing a ~2h burn down to approximately 8-15 min at the default 1500 mm/min, i.e. within ~2-3x of LightBurn's ≈5min rather than 24x.

Closing the final gap to ≈5min requires the user to actually run a higher feed (LightBurn-class, e.g. 6000 mm/min) AND the host to sustain it — which is where P1 #3 (decoupling the ack/send loop from the per-ack 200-row render+reflow) becomes load-bearing: without it the renderer-thread back-pressure would cap effective throughput at high feed and re-starve the planner (~1.5-4x penalty). With P0 #1, P0 #2, and P1 #3 all done and feed raised to match LightBurn, an estimated final burn time of ≈5-8 min (parity-class with LightBurn) is realistic. The remaining P2/P3 items are quality/edge-case polish and do not materially change wall-clock.

## Ranked bottlenecks (by analysis lens)

### Lens: STREAMING/PROTOCOL — send pacing: per-line ok-wait vs buffer saturation, status-poll blocking, per-line awaits, and host-side back-pressure on the renderer main thread. All code claims re-verified against the live tree at C:/Users/Asus/LaserForge-2.0.

**Timing model:** BACK-OF-ENVELOPE — can the streaming/protocol lens account for ~2h vs ~5min (24x)?

ASSUMPTIONS (all grounded in verified code + GRBL defaults):
- Fill burn line ~26 bytes incl. \n (verified format grbl-strategy.ts:95-98, fmt=3dp; e.g. 'G1 X145.678 Y167.890 S1000\n' ≈ 27B, 'G1 X45.678 Y67.890 S0\n' ≈ 22B). 4 lines/run (G0 + 3 G1).
- RX window = 120B (streamer.ts:21) → pipeline depth = floor(120/26) ≈ 4-5 lines in flight.
- Default fill feed 1500 mm/min = 25 mm/s; default hatch spacing 0.2mm; default 5mm overscan/side.
- A traced/halftone image fragments each scanline into many short runs. Job size 10^4-10^5 emitted lines is realistic (perf fixture models 35,000 contours, fill-hatching.perf.test.ts).

STEP 1 — How many lines/sec must the host sustain to keep GRBL's planner fed?
A short fill run (burn say ~2-5mm + 10mm overscan at 25 mm/s) executes its motion in ~0.4-0.6s. That run is 4 lines. So motion alone consumes only ~7-10 lines/sec — LOW. KEY INSIGHT: at the *default 1500 mm/min* fill feed, the required host line-rate is modest (tens of lines/sec), so the *streaming protocol is NOT the binding constraint at default speed*. The character-counting streamer (4-5 lines in flight) plus GRBL's own 16-block planner can easily supply 7-10 lines/sec even with a slow per-ack render. => At default feed, the dominant 24x cost is MOTION (per-run overscan-at-feed + full-stop accel/decel), NOT streaming. The streaming lens CANNOT by itself explain 24x at default feed.

STEP 2 — Where streaming COULD bind: high feed (matching LightBurn settings, e.g. 6000 mm/min = 100 mm/s).
At 100 mm/s a ~3mm short run executes in ~30-60ms incl. ramps. 4 lines per ~45ms ⇒ host must sustain ~90 lines/sec; denser detail (1-2mm runs) pushes toward ~150-250 lines/sec. Now per-ack host cost matters:
- Per inbound 'ok': handleLine does set({log}) (line 111) THEN advanceStream does set({streamer}) (line 168) = 2 synchronous Zustand commits ⇒ 2 React renders of LaserLog (subscribes s.log, not memoized) over up to 200 rows (filter line 27 + map 46-51) + 1 forced reflow (scrollHeight/scrollTop/clientHeight, lines 23). Renders run INLINE on the renderer main thread inside the Web Serial read loop (web-serial.ts:181 'for (const h of lineSubs) h(line)'), and the next bytes only go out from inside advanceStream (line 169) — so render time serializes with ack-consume-and-resend.
- If a saturated 200-row LaserLog render + reflow costs ~3-8ms, and each ok triggers ~2 of them, effective ack handling ≈ 6-16ms ⇒ ceiling ~60-160 lines/sec. That is right at/below the ~90-250 lines/sec a dense fill needs at 100 mm/s ⇒ planner starves, head decelerates between short runs ⇒ multi-x penalty STACKED ON the motion penalty.

STEP 3 — Verdict on the lens:
- Pure-protocol (per-line wait): REFUTED — it's character-counting (verified). No per-line wait, no artificial sleep/throttle, status poll is fire-and-forget and doesn't touch the streamer. So the classic GRBL 'send-and-wait' 24x story does NOT apply here.
- The lens's REAL contribution = host-side back-pressure: per-ack main-thread render+reflow throttling ack throughput, no worker/Electron offload (verified). This is a genuine multiplier ONLY in the high-line-rate regime (high feed / very dense detail). Magnitude is unprovable from static code (need a renderer profile + serial trace); plausibly 1.5-4x.

CONCLUSION: This lens can account for AT MOST a ~1.5-4x slice (and only when feed is high enough that required line-rate exceeds the per-ack-throttled ceiling). It CANNOT account for the full 24x on its own. The bulk of the 24x is the MOTION/EMIT model (per-run 5mm overscan traversed at cutting feed + forced full-stop accel/decel on thousands of fragmented runs — fill-overscan.ts:26-31, grbl-strategy.ts:95-98, planner.ts:239), which is outside this lens. Through the streaming lens specifically: ~5min→~10-20min is explainable; ~5min→~2h is not.

### Lens: MOTION / SEGMENT-DENSITY: the bottleneck is the number and shape of the emitted motion blocks for a Fill. A raster-traced image fragments every 0.2mm scanline into many short, separately-emitted hatch runs; each run is emitted as a fixed 4-block sequence (G0 rapid + 5mm lead-in at CUTTING feed + burn + 5mm lead-out at CUTTING feed), and the planner forces a full decel-to-zero / accel-from-zero at BOTH ends of every run (junctionVelocity returns 0 across every cut/travel boundary). The collapse in effective feed is therefore NOT GRBL planner starvation from streaming — it is host-side toolpath structure: per-run fixed overhead (overscan-at-feed + accel/decel triangle) multiplied by thousands of runs, with no merging or path-optimization to amortize it. Verified against live code at C:\Users\Asus\LaserForge-2.0.

## Fix plan

### Skip/shrink overscan on short fill runs and emit overscan as RAPID (G0), not cutting-feed G1  `P0` · effort M · expected ~3-5x on its own at default feed (collapses the dominant overscan-at-feed term; rapids at 6000 vs 1500 mm/min plus eliminated runway on short runs)
- **Targets:** Cause 1 (per-run overscan at cutting feed) — the single largest term
- **Approach:** In src/core/output/grbl-strategy.ts emitFillGroup and src/core/job/fill-overscan.ts: (a) emit the lead-in/lead-out as G0 (rapid, laser-off) instead of `G1 F{feed} S0` — they are already S0 laser-off moves so making them rapids is semantically identical for the beam but ~4x faster at default feed; and (b) add a length threshold so runs shorter than ~2× overscan get little or no runway (overscan exists to let the head reach feed before firing; on a sub-mm run the 10mm runway is pure waste). Mirror planner.ts appendFillGroupBlocks (lines 120-123) to price lead moves at travelV so the ETA stays honest. Keep a small lead only where the burn is long enough to benefit.
- **Modules:** src/core/output/grbl-strategy.ts, src/core/job/fill-overscan.ts, src/core/job/planner.ts
- **Risk:** Low-medium. Changing lead moves from G1 to G0 is the kind of change that touches g-code emission, so it needs a unit test asserting the new G0+S0 shape AND a planner test that ETA drops. Edge case: ensure the burn move itself still carries F (currently F is on the lead-in G1 line 96; if the lead-in becomes G0 the F must move onto the burn G1 at line 97).
- **Safety:** MUST keep S0 on every lead/rapid move (PROJECT.md invariant #3, laser-off-on-travel) — G0 lines already carry S0 in emitSegment; preserve that. Does not touch pause/feed-hold or e-stop paths. Hardware verification needed on the Falcon A1 Pro before marking Shipped (touches g-code emission).

### Merge consecutive collinear hatch runs into one continuous laser-on sweep with S0 blanking across gaps (raster-style fill emit)  `P0` · effort L · expected ~5-10x (eliminates thousands of full-stop accel/decel cycles AND the per-run G0; turns ~10k short runs into ~500 long row-sweeps, the same shape that gives LightBurn its ≈5min)
- **Targets:** Causes 2 + 3 (forced full stop per run; run-count explosion) — the structural fix that matches LightBurn
- **Approach:** Add a fill-emit pass that, for runs on the SAME scanline (same Y, already snake-ordered in fill-hatching.ts:233), emits ONE continuous G1 sweep across the row and blanks the inter-run gaps with S0 — exactly the model the raster path already uses (src/core/raster/emit-raster.ts:166-186 sweeps a whole row with S-modulation, overscan once per row edge). Either (a) generalize emitRow's continuous-sweep+S-blank approach for FillGroups in grbl-strategy.ts, or (b) add a consolidation step in compile-job before emit that joins adjacent same-Y runs into a single multi-segment polyline with explicit on/off S values. This removes the per-run G0 and the cut↔travel full stop between features within a row, so the head cruises the row instead of stopping at every interval. Update planner.ts appendFillGroupBlocks to model the continuous sweep (no junctionVelocity=0 between same-row features).
- **Modules:** src/core/output/grbl-strategy.ts, src/core/raster/emit-raster.ts, src/core/job/compile-job.ts, src/core/job/planner.ts

Requires careful tests that every gap span carries S0 and every ink span carries S{s}, and an integration test from compile-job through emit. - **Safety:** CRITICAL: the S0-blanking across gaps is the laser-off-on-travel invariant (#3) expressed inline — a regression here fires the beam over blank space. **Safety concerns:**
- Fix #3 UNDERSTATES a concrete, repo-specific concurrency hazard. The plan proposes 'send the next bytes FIRST (or batch the log/streamer state into a single throttled commit)' in advanceStream — i.e. defer/reorder the `set({streamer})` commit. But laser-store.ts resumeJob (lines 414-432), stopJob (434-444), and the onClose handler (283-297) all use functional `set((s) => ... step(... s.streamer))` SPECIFICALLY because (per the R-H2 audit comment at lines 416-424) reading a stale streamer snapshot and re-stepping can 'push more bytes than the buffer holds -> dropped commands -> uncontrolled head motion.' If Fix #3 batches or defers the advanceStream streamer commit, a concurrent resume/stop/disconnect functional-set can read a streamer state whose in-flight byte accounting has NOT yet been decremented by the just-arrived ack, recompute step(), and overflow GRBL's 120-byte RX window. This is exactly the failure the existing code was hardened against. Fix #3's safetyNote mentions 'must not drop or double-count an ack' generically but does NOT name this resumeJob/onClose interaction. Mitigation: keep streamer accounting a single synchronous source of truth (only the LOG may be batched/deferred; the streamer state commit and the in-flight byte math must remain synchronous and atomic with the ack). Add a failure-mode test: ack arrives concurrently with resumeJob; assert inFlightBytes never exceeds rxBufferBytes.
- Fix #3 part (1) ('send the next bytes FIRST, before the store commit') changes ordering on the live streaming/ack path. Currently advanceStream does onAck -> set({streamer}) -> safeWrite (laser-line-handler.ts:166-170), and on safeWrite rejection it sets disconnectStreamer(acked.state). If bytes are written before the streamer state is committed, an exception/disconnect between write and commit can leave the observable streamer state inconsistent with what was actually pushed to the wire (over- or under-counted in-flight), which on a laser is a dropped-command / overflow risk. Any reorder must preserve the invariant that the committed streamer state always reflects exactly the bytes already written, and the disconnect-on-write-failure path (line 170-172) must still fire.
- Fix #3's log-batching must not defer alarm/error CLASSIFICATION or the advanceStream call that fires on them. handleLine fires advanceStream immediately on alarm (line 143) and error (line 148), and onAck('alarm') transitions the streamer to 'cancelled' (streamer.ts:138-144) which halts further sends. The plan's safetyNote does call this out ('must still fire advanceStream immediately on alarm/error'), but since the proposed change batches set({log}) and the alarm/error branches ALSO call set({...}) before advanceStream, the implementer must ensure only the cosmetic log append is batched — the alarmCode/lastError state set and the advanceStream(ack) call must stay synchronous. A regression here would let the streamer keep feeding the planner after the controller has alarmed (e.g. hit a hard limit).
- Fix #2 (continuous-sweep fill with S0-blanking across gaps) is correctly flagged as the highest-bar safety change, but note a template mismatch the plan glosses: emit-raster.ts blanks gaps under M4 dynamic mode (comment lines 23-24,170) where S0 also guarantees a dark diode while moving; Fill runs under M3 constant power (preamble 'M3 S0', grbl-strategy.ts:37). Under M3, S0 is power-zero so the beam is dark, which is fine — but the emitter must emit an explicit S0 token on EVERY gap segment and re-assert S{s} on EVERY ink segment (G-code S is modal). A single missed S-reset after a gap would fire the beam across an interior hole at full power. The plan's 'test the exact S-value sequence per sweep' is the right control; make it an exhaustive per-segment assertion, not a spot check, and include a multi-hole contour (>=3 interior runs per scanline) fixture.
- Fix #1 has a real but plan-acknowledged correctness trap: today the burn G1 inherits modal F from the lead-in G1 at grbl-strategy.ts:96. If the lead-in becomes G0, F is no longer asserted before the burn, so the burn G1 at line 97 MUST gain `F{feed}` or the controller burns at whatever stale modal feed was last set (potentially a rapid-class or previous-group feed) -> wrong dwell -> over/under burn. The plan flags this ('ensure the burn move still carries F'); treat it as a required assertion in the unit test, and also confirm the planner's appendFillGroupBlocks is updated to price the lead moves at travelV (currently cutV, planner.ts:121,123) so the safety/ETA model stays honest.

### streaming-protocol
_The GRBL streamer itself is correctly implemented as a character-counting protocol (not call-response): step() packs as many whole lines as fit in a 120-byte RX-buffer window, and every ack tops it back up, so ~4-6 fill lines are in flight at once. The protocol is therefore NOT the bottleneck in isolation. The real defect is host-side send throttling caused by per-ack main-thread UI work: every inbound line (including every 'ok') runs synchronously on the renderer main thread and triggers TWO Zustand set() calls (log append + streamer advance), each of which forces LaserLog (mounted during the job) to re-filter and re-render up to 200 DOM rows and perform a forced layout reflow (scrollHeight/scrollTop/clientHeight reads). Because the Web Serial read loop, ack handling, and the next write all run on that same main thread (no worker, no Electron-side streaming offload), slow renders serialize with and throttle ack processing -> the GRBL planner buffer starves -> the head decelerates to a stop between short fill moves. That stutter, not a motion-planning difference, is what turns LightBurn's 5 min into ~2 hr. Polling is NOT the cause (it is refuted below)._

## LightBurn / GRBL research

### GRBL 1.1 streaming and buffer management: why a per-line "send and wait for ok" streamer makes a dense Fill engrave ~24x slower than LightBurn, and the techniques that close the gap
_A ~24x gap (2h vs ~5min) on a dense Fill engrave is not one bug but a stack of multiplicative streaming/buffer mistakes — and it is fixable mostly on the host side. The prime suspect is the streaming protocol: a naive engraver uses simple SEND-AND-WAIT-FOR-OK (one line, block until 'ok'), which serializes a ~1-10 ms USB round-trip onto EVERY short Fill segment AND lets GRBL's 16-block planner buffer drain between sends, so the machine does start-stop motion far below the commanded feedrate. LightBurn instead uses the CHARACTER-COUNTING protocol: it keeps GRBL's 128-byte RX buffer saturated with many lines in flight, pipelining away the latency and keeping the look-ahead planner full ('only the speed of the serial connection is the limit'). Stacked on that, a dense Fill is a worst case in three more ways: (1) GRBL's planner can only optimize the 16 blocks it currently holds, so tiny 0.1 mm segments give only ~3.2 mm of lookahead and the planner caps speed to what it can still stop within — LightBurn fixes this with CLUSTERING (one long G1 carrying up to 16 colon-delimited S/power values), which lengthens moves ~16x, cuts 'ok' round-trips ~16x, and shrinks bytes-on-wire (authors report ~4x greyscale gain, >20,000 mm/min); (2) at 115200 baud one-G1-per-pixel raster is bandwidth-capped near 6000 mm/min @ 250 DPI; (3) LightBurn's FAST-WHITESPACE skip and OVERSCAN cut travel/ramp time that a naive emitter wastes. Real-time bytes ('?','!','~',0x18, override bytes) bypass the line buffer and should be used async at <=5 Hz so status polling doesn't starve the motion stream. Concrete recommendation for LaserForge's GrblController/streaming path: replace any send-and-wait loop with character-counting flow control against RX_BUFFER_SIZE=128, query GRBL for its real buffer at connect, keep the 16-block planner fed, add fast-whitespace + overscan to the Fill plan, and (if the Falcon's firmware advertises [CLUSTER:n] in $I) emit clustered raster. Source for planner depth is gnea/grbl planner.h (BLOCK_BUFFER_SIZE=16) and the v1.1 Interface wiki._

- **Simple send-response (send-and-wait-for-ok) streaming — the likely root cause**: Host sends ONE G-code line, then blocks until GRBL returns 'ok' (or 'error:') before sending the next line. gnea calls this 'the most fool-proof and simplest method.' The fatal weakness for dense jobs is the per-line round-trip lag: 'there is a cumulative effect, because there is a lag with every G-code block sent to Grbl.' That lag = host-side write + USB/CH340 packetization + GRBL parse + ok transmit + host read. On a USB-serial diode/CO2 setup this is typically ~1-5 ms per line (CH340/FTDI USB latency timer alone is often 1-16 ms). Crucially, while the host waits for 'ok', it sends NOTHING, so GRBL's 16-block planner buffer drains. The wiki: 'the performance through these types of motions will never be full-speed, because the look-ahead buffer will always be partially full when using this streaming method.' A Fill engrave is exactly the pathological case — 'sequential, very short, line segments with high feed rates' — so the planner repeatedly empties and the machine does start-stop motion at a small fraction of commanded speed. — Dominant factor. If each pixel/segment is one short G1 move, motion rate is capped near 1 / (per-line latency). At ~3 ms/line that is ~330 moves/sec; at ~10 ms (bad USB latency) only ~100/sec. A 0.1 mm dot at 330 moves/s = 33 mm/s max, regardless of the feedrate you set — and with planner starvation the EFFECTIVE speed is lower still because the axis must decelerate to near-zero between starved blocks. Switching this single behavior to a buffered protocol commonly yields 3-10x on dense raster/fill. Accuracy: send-and-wait is the most reliable (no overflow risk) — the trade is purely speed.
- **Character-counting streaming protocol (keep the 128-byte RX buffer saturated)**: The host maintains a running count of characters it has sent but GRBL has not yet acknowledged, and sends as many whole lines as fit inside GRBL's serial RX buffer (RX_BUFFER_SIZE = 128 chars) WITHOUT waiting line-by-line. Each 'ok' tells the host one line cleared, freeing that many bytes to send more. The wiki worked example: lines of 25/40/31/58/20 chars — host fires the first three (96 chars) immediately, waits only when the next line won't fit in 128, then refills as 'ok's arrive. Effect: 'It always ensures Grbl's serial read buffer is filled, while never overflowing it... maximizes Grbl's performance by keeping the look-ahead planner buffer full by better utilizing the bi-directional data flow of the serial port.' Because multiple lines are always in flight, the round-trip latency is hidden (pipelined) instead of serialized, so the planner stays fed and can plan to full speed. This is the method LightBurn / ioSender / UGS / bCNC / LaserGRBL use, NOT send-and-wait. Caveat from the wiki: do NOT use it for EEPROM-writing settings ($$, $x=) because interrupts (incl. serial) are disabled up to ~20 ms during EEPROM writes and a char can be lost. — This is the biggest single lever after eliminating send-and-wait. gnea: 'we have stress-tested this character-counting protocol to extremes and it has not yet failed. Seemingly, only the speed of the serial connection is the limit.' Removes the per-line latency wall, so motion rate becomes bounded by serial bandwidth and planner CPU rather than round-trip time. Typical real-world result on raster/fill: several-fold throughput gain and elimination of stutter, with no loss of accuracy (the protocol is provably overflow-safe by construction). With ~12-15 short lines resident in the 128-byte buffer plus the 16-block planner, GRBL has enough lookahead to sustain commanded feedrate on dense work.
- **Keeping the 16-block planner (look-ahead) buffer full — why starvation throttles dense jobs**: GRBL's planner is a ring buffer of BLOCK_BUFFER_SIZE motion blocks — confirmed in grbl/planner.h as 16 (15 if USE_LINE_NUMBERS), comment: 'The number of linear motions that can be in the plan at any given time.' Acceleration/junction-velocity look-ahead can only optimize the moves currently IN that buffer. To enter a corner or a segment fast, the planner must already see enough downstream distance to guarantee it can still decelerate to a safe stop at the buffer's end (so the machine never overruns un-planned geometry). With tiny segments this lookahead distance is brutally short: per the FluidNC clustering analysis, 'If each move is a single pixel in a grayscale image, and pixels are 0.1 mm, that means you can only see 3.2 mm ahead' (16 blocks x 0.2 mm). At 100 mm/s with ~10 mm/s^2-class accel you simply cannot reach commanded speed within 3.2 mm, so GRBL caps velocity to whatever it can safely stop within the visible window. Send-and-wait makes this far worse because the buffer is usually only 0-1 blocks deep, not 16. — Sets a hard speed ceiling on short-segment work even with perfect streaming: ceiling ~ sqrt(2 * accel * lookahead_distance). Short segments + shallow buffer = the machine runs well below the feedrate you set and corners get rounded/slowed. On firmware you control, raising BLOCK_BUFFER_SIZE dramatically helps (grblHAL/FluidNC users report 48,000-70,000 mm/min raster with 1024-block buffers vs stuttering at 100 mm/s on stock 16); but on a stock Arduino-class GRBL 1.1 board the buffer is fixed at 16, so the practical fix is to make the SEGMENTS LONGER (clustering) and KEEP THE BUFFER FED (character-counting) rather than enlarge the buffer.
- **Serial bandwidth ceiling at 115200 baud — the second hard wall on raster/fill**: GRBL runs 115200 8N1 = 10 bits/byte = 11,520 bytes/sec max. A per-pixel raster line like 'X137.5S713\n' is ~10-12 bytes. Even with a perfectly pipelined stream, that caps you near 11520/11 ~ 1000 moves/sec. A LaserGRBL contributor computed exactly this: at 250 DPI (0.1 mm dots) that is '1000 x 0.1 mm/sec = 100 mm/sec = 6000 mm/min', and 'doubling resolution halves' it. So unclustered raster is bandwidth-bound to roughly 6000 mm/min at 250 DPI on USB — before any planner or latency loss. (Note: the grblHAL slow-raster discussion found lowering baud barely changed raster speed, meaning on stock boards the per-line/planner overhead usually bites BEFORE raw bandwidth — but at higher DPI bandwidth becomes the wall.) — Hard ceiling ~6000 mm/min @ 250 DPI, ~3000 @ 500 DPI for one-G1-per-pixel raster on USB. Quantitative and unavoidable without packing more pixels per byte. No accuracy impact; it is purely a throughput cap. This is why a naive emitter that writes one G1 per pixel can never match LightBurn on a dense image even if its streaming protocol were perfect.
- **G-code clustering (multiple S/power values per move) — LightBurn's $Cluster mode** _(not GRBL)_: Instead of one G1 per pixel, emit ONE longer G1 carrying many colon-delimited S (power) values, each applied to a sub-span of the move: e.g. 'G1 X0.961 S255.5:259.8:268.5:262:...' (Cluster16 packs 16 brightness values; original spec allowed 8). The controller advertises support via [CLUSTER:8] or [CLUSTER:16] in the $I response; each S is scaled by $30 as usual. Three simultaneous wins: (1) the planner sees ONE long move instead of 16 tiny ones, so lookahead distance jumps ~16x and the speed ceiling rises; (2) far fewer 'ok' round-trips, so per-line latency overhead drops ~16x; (3) fewer bytes on the wire (one set of coordinates amortized over many pixels). LightBurn enabled cluster mode in v1.2.02; supported by grblHAL and FluidNC, NOT by stock gnea GRBL 1.1. — Authors quote ~4x greyscale improvement: 'allows rastering >20,000 mm/min greyscale with smaller dot size which is likely 4x improvement.' Measured 100x100 mm @ 423 DPI: non-cluster USB 50.0 min vs cluster USB (LightBurn) 29.3 min (~1.7x there, accel-limited test); file size ~34 MB -> ~20 MB (~1.7x less data). Bigger gains at higher DPI/speed where the 16-block lookahead and per-line latency otherwise dominate. Accuracy is preserved (each sub-span keeps its own power); it is transparent to normal single-S G-code.
- **Real-time command bytes vs the line-based protocol (don't let status polling pollute the stream)**: GRBL splits its interface in two. Normal G-code goes line-by-line through the 128-byte RX buffer and consumes a planner block. A small set of single-byte REAL-TIME commands are intercepted the instant they arrive and NEVER enter the serial buffer or the planner: '?' (status report), '~' (cycle start/resume), '!' (feed hold), 0x18/Ctrl-X (soft reset), plus the 0x80+ extended-ASCII feed/rapid/spindle override bytes. This lets a GUI poll position/state and apply overrides mid-job without disturbing the streamed line flow. The wiki recommends polling '?' at NO MORE than 5 Hz ('10 Hz may be possible... at some point there are diminishing returns and you are taxing Grbl's CPU'). Relevant to LightBurn-vs-other gaps: a streamer that (a) waits for the '?' status reply inline, or (b) over-polls, or (c) sends '?' as a full line, steals CPU/bandwidth from the motion stream and worsens starvation. — Indirect but real on dense jobs: correct use (single byte, async, <=5 Hz) is essentially free; misuse (high-rate or blocking polling) measurably reduces sustainable move rate and re-introduces stutter. Reported feed-rate in '?' lags actual position by ~30-50 ms — fine for display, but don't gate streaming on it. No accuracy impact when used as designed.
- **Fast Whitespace skip + Overscan (LightBurn travel-time optimizations a naive Fill emitter omits)**: Two host-side scan optimizations LightBurn applies to Fill/Image that a from-scratch engraver often skips. Fast Whitespace: 'LightBurn will boost the speed through blank areas to the speed you indicate, if it is faster than the current engraving speed' — i.e. it rapids across empty spans of a scanline instead of crawling at burn speed. Overscan: adds short accel/decel moves OUTSIDE the image at the ends of each scan line so the axis reaches steady feedrate before firing (also evens edge darkness). These are explicitly offered for GRBL-class lasers as software substitutes for what DSP controllers do automatically. If your app burns blank regions at engrave speed and/or has no overscan, every scanline wastes time traversing whitespace and ramping inside the burn region. — Highly image-dependent: on a sparse image (lots of white) fast-whitespace alone can cut total time by 2x or more; on a solid-black fill it does little. Overscan trades a small amount of extra travel for correct (not throttled) feedrate across the burn, improving both speed consistency AND edge quality. Both are GRBL-applicable and are part of why LightBurn's wall-clock beats a naive emitter even at equal streaming efficiency.
- **Faster controller / larger buffers (the firmware-side escape hatch)**: When the bottleneck is planner CPU or buffer depth rather than the host protocol, moving to a faster MCU and/or larger buffers removes the wall. Reported data points: Arduino Uno (16 MHz, no FPU) ~1400 mm/min raster-class; Arduino Due / SAM3X8E (84 MHz) ~7000 mm/min (5x Uno); grblHAL on Teensy 4.1 (600 MHz ARM w/ FPU) >70,000 mm/min, comfortable 48,000 mm/min with a 1024-block planner buffer; grbl-LPC with 250-block planner + 10,240-byte serial buffer runs 'the most complex dithered rasters at feeds up to F15000 without stuttering.' The common thread: FPU + clock speed for the planner math, plus deep BLOCK_BUFFER_SIZE so short segments still get adequate lookahead. — 5x (Uno->Due) to 50x (Uno->Teensy) raster throughput in cited tests. But this is firmware/hardware, not something a host app like LaserForge can change on the user's Falcon A1 Pro stock board — listed for completeness. For a fixed stock GRBL 1.1 diode/CO2 board the actionable levers remain: character-counting streaming, clustering (if firmware supports it), and host-side scan optimizations.

### GRBL 1.1 motion planning and why short Fill segments collapse effective feed rate far below the set speed (the 24x slowdown mechanism)

- **Trapezoidal velocity profile and the segment-length-vs-speed limit (the core slowdown mechanism)**: GRBL never jumps to a feed rate; every move is a trapezoid: accelerate at $12x up to the lesser of the set feed and the axis max rate ($11x), cruise, then decelerate. The distance to ramp from speed v_entry to v_target under acceleration a is d = (v_target^2 - v_entry^2) / (2a) (the kinematic identity v^2 = u^2 + 2ad). If a segment is too short to finish accelerating before it must start decelerating, the cruise phase vanishes and the profile becomes a TRIANGLE: peak speed is capped at v_peak = sqrt(a * d) for a symmetric accel/decel from rest over distance d. CONCRETE NUMBERS: a raster/Fill row is decomposed into many tiny laser-on/laser-off line segments (often << 1 mm each at image resolution; e.g. 254 DPI = 0.1 mm/pixel). With acceleration a = 1000 mm/s^2 and a 0.5 mm segment from rest, v_peak = sqrt(1000*0.5) = 22 mm/s; for 0.1 mm, v_peak = sqrt(1000*0.1) = 10 mm/s. If the target Fill speed is 167 mm/s (10000 mm/min), the machine reaches only ~6-13% of it -> a 8x-16x time penalty purely from geometry. With the GRBL DEFAULT a = $120 = 10 mm/s^2, to even reach 167 mm/s needs d = 167^2/(2*10) = 1395 mm of straight runway (and the same to stop) — utterly unattainable in short Fill segments, so the head crawls. This is the dominant contributor to a 24x slowdown. — Largest single factor. Effective feed on short segments scales as sqrt(a*d), not the set feed. Raising acceleration helps as sqrt(a): going from a=300 to a=1200 (4x) only doubles achievable speed on a fixed short segment. The real fix is making segments longer (continuous on/off within one long sweep + overscan) so a single trapezoid spans the whole row and actually cruises at the set feed — this alone can recover most of a 10x-20x loss.
- **The 15-16 block look-ahead planner buffer and the decelerate-to-stop-at-buffer-end constraint**: GRBL holds only BLOCK_BUFFER_SIZE motions in its planner (defined as 15 with USE_LINE_NUMBERS, else 16; the wiki rounds to '~17'). It runs a reverse pass (compute junction speeds backward) then a forward pass (cap by accel from entry speed), maximizing the velocity profile across exactly those buffered blocks. CRITICAL: the planner cannot see past the last block, so it must assume the machine comes to a complete REST at the end of the buffer — verbatim from planner.c: 'The last (or newest appended) block is planned from a complete stop (an exit speed of zero).' planner.c then states the slowdown mechanism directly: 'NOTE: Since the planner only computes on what's in the planner buffer, some motions with lots of short line segments... may seem to move slow. This is because there simply isn't enough combined distance traveled in the entire buffer to accelerate up to the nominal speed and then decelerate to a complete stop at the end of the buffer.' So the SUM of distances of the ~15 buffered short segments is the runway. If 15 segments x 0.3 mm = 4.5 mm total, peak speed over that runway at a=1000 is sqrt(1000*4.5) ~= 67 mm/s, and it must come back to 0 at the end — i.e. it can never sustain a high feed when segments are tiny, no matter how fast the data arrives. — Hard ceiling independent of host streaming speed. The shorter the segments, the more the 15-block horizon clamps top speed. A larger buffer (e.g. ESP32 GRBL_ESP32 / FluidNC, or grblHAL with bigger BLOCK_BUFFER_SIZE) raises the horizon and the achievable speed on dense art; classic 8-bit AVR GRBL is stuck near 15-16.
- **Serial-buffer starvation / streaming throughput limit (the 'stutter')**: Separate from the planner math: the host must feed g-code lines faster than GRBL executes them, or the planner empties and the head does visible start-stop 'stutter' that drops effective feed further. GRBL has a 127-byte serial RX buffer feeding the ~16-block planner. Interfacing-with-Grbl (verbatim): with naive send-response streaming, 'some motions with lots of short line segments at high feed rates... the cumulative lag can be large enough to empty and starve the look-ahead planner buffer... This could lead to start-stop motion when the streaming can't keep up.' It blames USB: 'Arduinos and most all micro controllers use a USB-to-serial converter chip that... like USB packet buffering and delays... can wreak havoc to a streaming protocol.' The fix is character-counting streaming: keep a running count of bytes sent, subtract on each 'ok', keep the 127-byte RX buffer full without overflow. CONCRETE: a Fill row at 254 DPI over 100 mm = ~1000 tiny segments; if each g-code line is ~15-25 bytes, that is ~15-25 KB per row that must stream faster than the (already slow) motion consumes it. If the app sends one line and waits for 'ok' (round-trip latency 1-20 ms per line over USB), throughput collapses to tens of lines/s and the planner runs dry — exactly the LightBurn-forum stutter reports (XTool D1 'shaking' above ~105 mm/s; a user's 2-min file took >10 min). — Can independently cause a multi-x slowdown plus banding. Character-counting (or block-batching, e.g. the forum user's 32-line/1024-byte batches) restores throughput. WiFi/SD upload (send whole job to controller, e.g. Falcon/XTool WiFi) eliminates the serial bottleneck entirely. Note: this is necessary but NOT sufficient — even with a perfectly fed buffer, the trapezoidal/segment-length and 15-block constraints above still cap speed.
- **Junction deviation ($11) and the cornering / direction-reversal speed cap**: At each corner GRBL models a circular arc tangent to both segments and bounds the corner speed by centripetal acceleration: v_junction = sqrt(a * R), where the cornering radius is set indirectly by junction deviation delta via R = delta * sin(theta/2) / (1 - sin(theta/2)); theta is the included angle from the dot product of the velocity vectors. The corner speed is then min(v_junction, v_entry, v_exit). Consequence: a 90-degree turn gives a small R and a low v_junction; a full 180-degree reversal (theta -> 0) gives R -> 0 and v_junction -> 0, forcing a near-complete stop. RELEVANCE TO FILL: bidirectional raster turns the head around at the END of every scan line — a 180-degree reversal — so without overscan the head must decelerate to ~0 at each row end and re-accelerate, compounding the short-segment penalty across every line. Default $11 = 0.010 mm; higher = faster corners but more ringing/lost steps, lower = slower, more careful corners. — Dominates time only at row turnarounds and on jagged vector paths, not inside a smooth long sweep. Raising $11 (e.g. 0.01 -> 0.05-0.1 mm) speeds cornering at the cost of accuracy/ringing. The bigger win for Fill is OVERSCAN (LightBurn adds lead-in/lead-out so the reversal happens off the artwork and the row itself stays at full feed) and Fast Whitespace Scan (boost over blank gaps). These keep the firing portion at the set feed instead of ramping through it.
- **Acceleration $120/$121/$122 — the ramp-rate setting (units, defaults, multi-axis coupling)**: Per-axis acceleration in mm/sec^2. Wiki: 'A lower value makes Grbl ease slower into motion, while a higher value yields tighter moves and reaches the desired feed rates much quicker.' This 'a' is the term in d = v^2/(2a) and v_peak = sqrt(a*d), so it directly sets how short a segment can be before speed collapses. The STOCK default is a punishing $120 = 10 mm/s^2; real diode/CO2 gantries run far higher (commonly 500-3000 mm/s^2 for X on a light diode head). Multi-axis note: a diagonal move accelerates only as fast as the slowest contributing axis allows, so a low $121 (Y) drags X down on angled segments. Tune by ramping G0 moves until the stepper stalls, then back off 10-20%. — Effective short-segment speed scales as sqrt(a). Moving from the default 10 to ~1000 mm/s^2 is a ~10x increase in 'a' = ~3.2x more achievable speed on the same short segment and far less stutter — often the single highest-leverage GRBL setting for Fill. Too high causes mechanical ringing/wobble (visible banding) and lost steps; that is the accuracy ceiling. If one app shipped with stock $120=10 and the user (or LightBurn) was running a tuned $120, that mismatch alone explains a large chunk of a 24x gap.
- **Max rate $110/$111/$112 — the per-axis speed ceiling that can silently override the set feed**: Per-axis max velocity in mm/min, and also the G0 rapid rate. Wiki: 'Whenever Grbl plans a move, it checks whether or not the move causes any one of these individual axes to exceed their max rate. If so, it'll slow down the motion to ensure none of the axes exceed their max rate limits.' So even a long, well-fed segment is hard-capped: if the requested Fill feed is 10000 mm/min but $110 = 6000, the head physically tops out at 6000 regardless of what the job asks. Stock default is a very low 500 mm/min. Tune like acceleration: ramp until stall, back off 10-20%. — A pure ceiling, not a ramp. If the app's requested feed exceeds $110, you silently get $110 instead — a fixed-ratio slowdown (e.g. asking 10000 with $110=2500 = 4x slower) with zero error message. Setting it well above the highest feed you use removes it as a bottleneck; setting it too high risks stalls/lost steps. Check $$ output: a low $110/$111 is a common, easily-missed cause of 'it ignores my speed.'
- **Why LightBurn is fast: long continuous scan lines, overscan, Fast Whitespace, scanning-offset, and aggressive buffering**: LightBurn sidesteps every limit above by changing the GEOMETRY and the STREAM, not the physics. (1) It emits Fill rows as long continuous sweeps with power modulated inline (M4 dynamic/constant-power mode), so one trapezoid spans the whole row and actually CRUISES at the set feed instead of being chopped into sub-mm triangular segments — collapsing thousands of short blocks into a few long ones that the 15-block planner can sustain. (2) Overscan adds lead-in/lead-out so accel/decel and the 180-degree turnaround happen OFF the artwork, keeping the engraved span at full feed. (3) Fast Whitespace Scan boosts speed across blank gaps. (4) Scanning Offset Adjustment compensates for the beam-vs-motion lag that otherwise forces users to slow down. (5) Character-counting streaming (buffer size detected at connect) keeps the 127-byte RX buffer full so the planner never starves. An app that instead emits per-pixel on/off micro-segments, streams send-response, and/or relies on stock $110/$120 will hit all the penalties simultaneously — short-segment triangular capping x buffer-horizon clamp x serial starvation x low ceilings — which multiply into the observed ~24x (e.g. ~5 min vs ~2 h). — This bundle is where the 24x lives. Largest recoverable wins, in order: (a) emit long continuous Fill sweeps with inline power + overscan instead of short per-pixel segments (can be 10x+); (b) character-counting/batched or WiFi-SD streaming to stop stutter (multi-x); (c) ensure $120 acceleration and $110/$111 max rate are tuned, not stock (2-4x). Accuracy is preserved because overscan/scanning-offset move the compromises off the artwork rather than degrading the engraved region.

### How LightBurn makes Fill/scan engraving fast — techniques and why a naive GRBL sender can be ~24x slower
_A 24x slowdown (≈2h vs ≈5min) on the same image/size/settings is almost never the laser hardware or the line interval — it is the toolchain's motion/output design. LightBurn's speed on Fill/scan comes from a stack of cooperating techniques, but for a GRBL diode/CO2 setup the dominant factors are (1) NOT emitting per-pixel G-code: LightBurn generates long contiguous scan-line segments and (for grayscale/dithered) packs up to 16 power values onto a single long G1 move via "GCode Clustering," which collapses file size ~10x and keeps the controller fed; and (2) keeping the GRBL planner buffer (only 16 blocks, 15 with line numbers, on an ATmega328) from starving. When a sender emits one short G1 per pixel/dot, the 16-block lookahead cannot accumulate enough distance to reach the commanded feed, so GRBL decelerates — an Arduino-class board physically capable of 20,000 mm/min is measured running raster at ~7,000 mm/min, and at 0.5 mm dot spacing a commanded 18,000 mm/min collapses to ~8,569 mm/min. Layer those slowdowns over every line of a tall image and you get the multi-hour result. Secondary multipliers LightBurn applies: bidirectional scanning (2x), Fast Whitespace Scan (a measured 1:20→0:37, ~2.2x, on whitespace-heavy art), overscan to hold constant head speed (default 2.5% of speed) so it never has to slow the burn at line ends, line-interval maximization, trimming trailing white pixels, and scan-ordering (all-shapes-at-once vs individually vs flood-fill) to cut travel. GRBL laser mode ($32=1, M4) is the firmware-side enabler: it removes the spindle-style stop-on-power-change pause so consecutive S-value changes stream without halting, and dynamically scales power with speed so accel/decel zones stay even. For LaserForge: audit whether the Fill output is emitting one short move per scan-dot/pixel and changing S on every move (the classic 24x trap), whether it streams fast enough to keep ≥16 blocks queued, whether it uses M4 + $32, and whether it overscans + goes bidirectional. Clustering support depends on firmware (grblHAL/FluidNC/Smoothie), but every other technique applies to stock GRBL 1.1._

- **Long contiguous scan-line segments instead of per-pixel G-code (the primary 24x lever)**: For a Fill, LightBurn rasterizes each scan row and emits the longest possible G1 moves rather than one tiny move per pixel/dot. For solid black it is effectively a single long G1 across the run; for runs of equal intensity it does run-length encoding (one move covers many same-intensity pixels). A naive engine that instead emits one short G1 per pixel (and changes S on each) produces a file with orders of magnitude more, much shorter, motion blocks. Each short block carries serial + parse + planning overhead, and — critically — is too short for GRBL's lookahead to plan a high cruise speed (see planner-buffer entry). — This is the single biggest contributor to large (10x+) discrepancies. Per-pixel emission both bloats the stream (clustered LightBurn files were measured at ~1/10 the size of unclustered: ~20MB vs ~36MB) and forces the controller below commanded feed. No accuracy cost to using long segments where intensity is constant; quality is governed by line interval, not segment length.
- **GCode Clustering (pack up to 16 power values onto one long G1 move)** _(not GRBL)_: For grayscale/dithered fills where S must vary along a row, LightBurn condenses many per-pixel power changes into one longer move whose S values are listed colon-delimited, e.g. 'G1 Y-0.48 S0.875:0:0:0:0.875:0:0:0.875'. One move carries up to 16 brightness values (originally 8, limited by Smoothieware RAM; firmware advertises support via [CLUSTER:16] in the $I response). Each S is still scaled by $30 exactly like normal GRBL. This was created by LightBurn's author to beat Smoothie's CPU/bandwidth limit and now serves the same role on grblHAL/FluidNC. — Cuts the command/bandwidth count ~10x, lets the planner 'see farther ahead' so it re-plans less and can hold speed; measured ESP32 grayscale jobs dropped from 39.2 min (SD) / 50.0 min (USB) to 26.3 / 29.3 min (~30–35% faster, up to ~50% on deep-grayscale-with-sparse-S work) while preserving grayscale and diagonal engraving. No quality loss.
- **Keeping the GRBL planner buffer fed (avoid lookahead starvation on short segments)**: GRBL plans acceleration only over the moves currently in its planner buffer — BLOCK_BUFFER_SIZE = 16 blocks on the ATmega328 (15 if USE_LINE_NUMBERS is enabled, per planner.h). To cruise at the commanded feed it must fit accel-up + cruise + decel-down inside the queued distance. When moves are very short (per-pixel/per-dot), the ~16-block window never accumulates enough distance, so GRBL never reaches the programmed rate and decelerates instead. The fix LightBurn implicitly uses is fewer/longer moves (long segments + clustering) plus streaming fast enough that the buffer stays full; raising acceleration ($120/$121) or buffer size also helps but is bounded by RAM. — Quantified repeatedly: an Arduino Due (raster) capped ~7,000 mm/min though the machine could move 20,000 mm/min; at 0.5 mm dot spacing a commanded 18,000 mm/min fell to ~8,569 mm/min; vertical-bar tests dropped 17,119→12,119 mm/min as spacing went 2mm→1mm. These compound per line, easily yielding multi-x to >10x total. Larger-RAM ports (Teensy 4.1, planner 1024) reach ~48,000+ mm/min on the same raster.
- **Sender throughput / serial bandwidth (don't let the controller wait for commands)**: Even with good G-code, if the sender can't deliver bytes fast enough the controller drains its buffers and pauses to a momentary stop between moves — visible as stutter and an effective speed far below commanded. LightBurn streams with character-counting flow control and keeps the buffer saturated. ATmega328 GRBL has only a 128-byte serial RX buffer, so per-pixel streams saturate the link; bigger ports help (Grbl-Mega 256B, Grbl-LPC 10240B). A community fix for a stuttering xTool (CH340) was to batch up to 32 commands and hold a 1024-byte in-flight window before waiting for 'ok', which removed the stutter at high speed. — When throughput-bound, observed stutter onset at ~163 mm/s (xTool D1 Pro) with clean running ≤110 mm/s — i.e. a ~1.5x ceiling purely from I/O, on top of any planner-starvation loss. 115,200 baud / ~200 Kb/s USB is the practical ceiling for dense rasters on AVR.
- **GRBL laser mode ($32=1) — eliminate the stop-on-power-change pause**: Stock GRBL pauses motion whenever the S spindle value or M3/M4/M5 state changes, because a real milling spindle needs spin-up time. With $32=1 (laser mode), GRBL updates the PWM pin instantaneously and moves continuously through consecutive motion commands that carry new S values — no halt between them. This is what makes varying-power scan rows (and clustered S-lists) stream without a dead stop at every intensity change. G0 rapids and probing auto-disable the laser. — Without $32=1, every S change in a grayscale row can force a stop — catastrophic for raster (could alone explain a large multiple of slowdown plus burn marks at each stop). Enabling it is mandatory for fast/clean engraving; correctness benefit (no overburn at stops) as well as speed.
- **M4 dynamic laser power (speed-proportional power)**: M4 scales PWM power with the current speed relative to the programmed feed: full power at full speed, proportionally less while accelerating/decelerating, and laser fully off when stationary. This keeps deposited energy-per-unit-length constant through accel/decel zones. It pairs with laser mode and lets the engine push higher feeds (and rely on accel zones) without scorching ends. — Enables running closer to the limit and using shorter lead-in distances without edge overburn; primarily an accuracy/consistency win that indirectly supports higher speed. Safety bonus: laser off when not moving (won't burn through on a stall), unlike M3.
- **Overscanning (keep the head at constant speed across the burn)**: LightBurn adds extra travel past each line end with the laser off, so acceleration/deceleration happens outside the artwork and the laser fires only while the head is at the steady commanded speed. Distance is a percentage of speed (default 2.5%: e.g. 100 mm/s → ~2.5 mm of overshoot); the dialog shows the computed distance for your Speed × Overscan%. DSP/Ruida controllers do this automatically; on GRBL/g-code it is a user setting and matters most with bidirectional scanning. — Doesn't itself raise top speed — it lets you safely USE a high speed without darker edges/lighter centers. Costs a little extra travel per line that grows with speed (at very high speeds the overscan tax becomes visible: one example added ~3 min going from 100 to 400 mm/s). Net enabler of high-speed bidirectional fills with even tone.
- **Bidirectional scanning (fire on both passes)**: The head sweeps left-to-right firing, then right-to-left firing, instead of firing one way and dead-heading back. Combined with overscan to keep speed constant at the turnarounds. — Up to 2x faster (LightBurn: 'usually preferred, because it is twice as fast') by eliminating the non-burning return pass. Accuracy caveat: without correct Scanning Offset Adjustment it can ghost/double-image at high speed, because the head lags the commanded position differently in each direction.
- **Fast Whitespace Scan (jump blank gaps at high speed)**: Normally the whole image is traversed at the slow burn speed. With Fast Whitespace enabled, LightBurn boosts the head to a separate, faster speed across blank spans (when that speed exceeds the burn speed), then drops back to burn speed when it hits ink — so empty regions aren't crawled over at engrave speed. — Large on whitespace-heavy art: a user job went 1:20 → 0:37 (~2.2x). Proportional to how much of the image is empty; no quality impact on the burned regions. (Firmware-availability caveat on some setups.)
- **Trim trailing/leading white pixels per scan line**: For each row, LightBurn stops the scan at the last inked pixel instead of running to the image's bounding box, and starts at the first inked pixel — the head doesn't travel into blank margin beyond content. This is distinct from Fast Whitespace (which speeds up interior gaps); trimming removes the end-of-line dead travel entirely. — Cuts both travel time and file size, especially for non-rectangular subjects on a large canvas. No quality cost. Compounds with overscan (only the trimmed extent gets the overscan margin).
- **Line interval maximization (fewest scan rows for the quality target)**: Line interval is the spacing between scan rows (inverse of lines-per-inch / DPI); it equals CNC stepover. LightBurn (Interval/Material test) guides you to the largest interval whose lines just touch without gaps. Job time scales ~linearly with row count, so widening the interval to the touch point removes rows. — Direct linear time lever: halving DPI (doubling interval) roughly halves the burn — 'reduced runtimes ... with no loss in quality' up to the touch point (may need a small power bump). Beyond the touch point you get visible gaps; below it you waste time over-overlapping. NOTE: same interval in both apps means this is NOT the cause of a 24x gap — look at segment/throughput design instead.
- **Scan ordering: all-shapes-at-once vs individually vs flood fill (minimize travel)**: LightBurn offers Fill-all-at-once (one set of full-width sweeps over everything in the layer — fewest direction changes, best for fast lasers ~300+ mm/s), Fill-shapes-individually / groups-together (finish a region before moving on — fewer long traverses, better for slow or fast-accelerating machines with spread-out shapes), and Flood Fill (prefers direction changes over long traversals, saving time when rapids are slow; can look erratic and is sensitive to backlash/tuning). The planner also uses inside-first ordering + nearest-neighbor for travel. — Travel-time optimization; magnitude depends on layout and rapid speed (can be tens of percent on sparse, spread-out designs). Wrong choice for the machine class can be slower; flood fill trades robustness for travel savings and can leave gaps on complex art if the machine isn't tight.

### M3 (constant) vs M4 (dynamic) laser power and GRBL laser mode $32: why constant power + stop-start is ~24x slower and scorches, and how M4 + $32=1 keeps the head moving continuously while scaling power with actual speed
_The ~24x Fill-engrave slowdown is almost certainly caused by GRBL NOT being in laser mode ($32=0), which forces a decelerate-to-zero stop at every spindle/S (power) change. The gnea/grbl v1.1 Laser Mode wiki states: 'Every time a spindle state M3 M4 M5 or spindle speed Sxxx is altered, Grbl would come to a stop, allow the spindle to change, and then continue.' A grayscale/dithered Fill emits a new S value per pixel/segment, so the head stops and re-accelerates thousands of times per scan line, never reaching cruise feed — turning a 5-minute job into hours. Setting $32=1 makes GRBL 'move continuously through consecutive motion commands when programmed with a new S spindle speed' without stopping. Layered on top: M3 constant power holds full power through the unavoidable accel/decel at line ends, scorching edges/corners and forcing time-costing overscan/lead-ins; M4 dynamic power instead scales PWM by actual_speed/programmed_rate (source: stepper.c `rpm *= prep.current_speed * prep.inv_rate`, inv_rate = 1/programmed_rate; spindle_control.c maps rpm->PWM via pwm_gradient), giving even tone at high feed and turning the beam off when stopped. LightBurn defaults Fill to Variable Power (M4) for exactly this reason and reserves Constant Power (M3) for cutting. LaserForge-specific caveat: even with $32=1 on the controller, if the app's GrblStrategy injects laserOff/laserOn/dwell or a fresh decel between each fill segment (instead of consecutive moves carrying inline S words and keeping GRBL's ~16-block look-ahead/127-byte buffer full), it reproduces the stop-start in software. The investigation should diff LaserForge's emitted g-code and serial pacing against LightBurn's for the same job, and confirm $32, before patching — per the repo's verify-against-live-code rule. Fastest fix path: $32=1 + M4 for fills + continuous-emission + overscan + machine-achievable feed/accel ($120/$121)._

- **Enable GRBL laser mode ($32=1) — the single largest speed lever; this is almost certainly the root cause of the 24x slowdown**: This is the dominant fix. In default (milling/spindle) GRBL, the controller treats every spindle command exactly like a real router spindle that needs RPM spin-up time. The gnea/grbl v1.1 Laser Mode wiki states it plainly: 'Every time a spindle state M3 M4 M5 or spindle speed Sxxx is altered, Grbl would come to a stop, allow the spindle to change, and then continue.' A Fill/raster engrave is the pathological case: grayscale/dithered fill emits a NEW S (power) value on essentially every pixel or short segment along each scan line — so with $32=0 the head decelerates to a full stop and re-accelerates thousands of times per line. The grbl laser_mode.md doc: 'if a laser starts and stops like this for every spindle change, this leads to scorching and uneven cutting/engraving!' Enabling laser mode ($32=1) changes the planner so 'Grbl will move continuously through consecutive motion commands when programmed with a new S spindle speed (laser power)' without stopping. At the planner/stepper level (planner.c / stepper.c), laser mode removes the forced decelerate-to-zero at spindle-speed changes and instead modulates only the PWM duty cycle on the fly while the head keeps moving. LightBurn's own GRBL troubleshooter tells users with travel/quality problems to 'type $32=1 in the console and press enter'; LaserGRBL similarly defaults to expecting laser mode. NOTE for the LaserForge engine: even with $32=1 on the controller, if LaserForge's own Plan/Output layer inserts a stop, a dwell, or a separate G1 block with its own decel between each power change (rather than emitting consecutive linear moves that each just carry an updated S word inline), it will reproduce the stop-start behavior in software regardless of the firmware setting. The fix has two halves: (a) the controller must be in laser mode, and (b) the g-code GrblStrategy emits must let consecutive fill segments flow as back-to-back moves with inline S changes, not stop/laserOff/laserOn per segment. — Largest single factor. Each unnecessary stop costs the full decel + accel time plus (in spindle mode) an additional dwell waiting for 'spindle spin-up'; over thousands of S changes per fill this dominates total job time. This is the most plausible explanation for a 2h-vs-5min (~24x) gap: 5 min in LightBurn implies the head is sustaining its programmed feed across each scan line, while 2h implies it is grinding to a halt at nearly every pixel. Quality also improves — eliminates the per-change burn spots/scorching the docs describe. Speed gain is workload-dependent but for high-S-density fills can easily be an order of magnitude or more.
- **M4 dynamic (variable) power — scales PWM by actual_speed / programmed_rate so corners and accel/decel ramps don't over-burn**: M4 (spindle-CCW) in laser mode is GRBL's dynamic power mode. Instead of holding a fixed duty cycle, GRBL continuously scales laser power by the ratio of the head's instantaneous speed to the programmed feed rate, so delivered energy per unit length stays roughly constant even while accelerating, decelerating, or rounding a corner. The exact mechanism is in the source: in stepper.c, when a planner block is flagged dynamic (PL_COND_FLAG_SPINDLE_CCW / is_pwm_rate_adjusted), GRBL precomputes the inverse of the block's programmed rate (prep.inv_rate = 1.0/pl_block->programmed_rate) and then per motion segment does rpm *= (prep.current_speed * prep.inv_rate) before prep.current_spindle_pwm = spindle_compute_pwm_value(rpm). So PWM = full_power * (current_speed / programmed_rate), clamped to the PWM range. spindle_control.c maps rpm to duty via pwm_value = floor((rpm - rpm_min)*pwm_gradient) + SPINDLE_PWM_MIN_VALUE, where pwm_gradient = SPINDLE_PWM_RANGE/(rpm_max - rpm_min). When the head is stopped, current_speed is 0 so power is 0 (M4 also forces rpm=0 when not moving), which is why M4 'turns off the laser when not in motion' and can't burn a hole in the bed if you forget M5. The grbl wiki summary: M4 'automagically scales power based on current speed related to programmed rate, so you can get super clean and crisp results, even on a low-acceleration machine.' LightBurn calls this 'Variable Power' and makes it the DEFAULT: 'ramps the power level of the laser up and down with the speed, producing a more consistent burn.' — Primarily an accuracy/quality lever, but it is what makes high feed rates usable: because power tracks speed, you do not have to slow the whole job down to avoid scorched corners and dark line-ends. Without it (M3 at high feed) the only ways to avoid edge over-burn are to add overscan distance or lower speed/power — both of which cost time or quality. M4 lets the planner keep the programmed feed and simply dim the beam through the unavoidable accel/decel zones, giving even tone with no time penalty. LightBurn explicitly recommends variable/dynamic power for engraving and image fills.
- **M3 constant power — why it is slower and scorches for fills; correct niche is cutting**: M3 (spindle-CW) holds the commanded power 'regardless if the machine is moving, accelerating, or stopped' (grbl laser_mode.md). For a fill/engrave this is bad on both axes. Quality: during the unavoidable decel at the end of each scan line and accel at the start, the head is slower but the beam is at full commanded power, so the edges receive more energy per mm than the middle — the classic 'darker at the edges, lighter in the middle' artifact LightBurn's Overscanning doc attributes to 'the laser slowing down to change scanning direction,' and the corner-scorch the docs warn about. Speed: to make constant power look even, you must compensate — the grbl doc says for M3 'it's a good idea to add lead-in and lead-out motions around the line... to give some space for the machine to accelerate and decelerate,' and LightBurn's equivalent is Overscanning (extra travel beyond the image with the beam off so it reaches full speed before firing). Those extra moves add distance/time that M4 doesn't need. M3's legitimate use is CUTTING, where you want constant cut energy and the path has few/long segments; LightBurn and FoxAlien both summarize it as 'constant power for cutting, dynamic power for engraving.' Using M3 for a dense fill is the wrong tool and contributes to both the scorching and the need for time-costing overscan/lead-ins. — For fills: worse quality (edge/corner scorch) AND slower once you add the overscan/lead-in moves required to hide that scorch. The slowdown from M3-vs-M4 alone is modest compared to the $32 stop-start effect, but M3 plus per-segment stops is the worst-case combination and compounds the problem. For cutting, M3 is correct and has no penalty.
- **Continuous motion through the planner/segment buffer (look-ahead) — what 'no stop at every boundary' actually buys**: GRBL's planner keeps a look-ahead buffer of upcoming blocks (BLOCK_BUFFER_SIZE, typically 16-17 blocks on an UNO/Nano-class AVR build) and a downstream segment buffer the stepper ISR consumes. With look-ahead, GRBL computes junction speeds between consecutive moves so it only decelerates as much as the geometry/junction-deviation ($11) requires, and otherwise carries velocity across segment boundaries. Laser mode is what lets a mere S (power) change stay inside this continuous flow instead of forcing an artificial full stop. Two practical consequences for the LaserForge plan/output design: (1) The g-code must be emitted as a stream of consecutive moves so the buffer can look ahead — if the sender waits for an 'ok' round-trip and lets the buffer drain between every short fill segment, the head stalls even with $32=1; GRBL's character-counting / 127-byte serial buffer protocol exists precisely to keep the planner fed (the LaserForge GrblController already models the 127-byte buffer). (2) Real machines never reach the programmed feed on very short fill segments anyway because of accel limits ($120/$121); references note real engraving loses 50-70% of nominal speed to accel/decel on short/curved paths. So the optimizer should prefer long uninterrupted scan lines (and overscan) over many tiny stop-prone segments. The 24x case is consistent with the head never getting to cruise speed because it is restarting every segment. — Keeping the buffer full and motion continuous is the difference between sustaining cruise feed across a scan line vs. accel/decel sawtoothing at near-zero average speed. Look-ahead buffer depth (~16 blocks AVR; more on 32-bit grblHAL/ESP32 forks) sets how far GRBL can plan; deeper buffers and higher accel ($120/$121) raise the average speed actually achieved. Combined with $32=1 this is what closes most of the gap to LightBurn.
- **Practical recipe to close the gap (verify, don't assume): $32=1 + M4 + overscan + sane accel/feed**: Concrete, ordered checklist grounded in the sources: (1) Confirm controller state — send $$ and check $32; if $32=0, set $32=1 (gnea/grbl wiki; LightBurn GRBL troubleshooter). (2) Use M4/Variable (Dynamic) Power for the Fill layer, not M3/Constant — LightBurn defaults Fill to Variable Power; reserve M3 for Line/cut layers (LightBurn Shared Settings; FoxAlien). (3) In LaserForge's own pipeline, verify the GrblStrategy emits consecutive fill segments as back-to-back moves with inline S words and does NOT inject laserOff/laserOn/dwell or a fresh decel between every pixel — a software-side stop-start will mimic $32=0 even with the firmware correct. This is the LaserForge-specific thing to audit, since the symptom (2h vs 5min on identical hardware/settings) points at the g-code the app generates or how the sender paces it, not the laser. (4) Add overscan (LightBurn Overscanning) so the head reaches programmed feed before firing — prevents edge scorch without slowing the engraved region. (5) Sanity-check accel ($120/$121) and feed: LightBurn notes if you command a feed the machine can't reach 'it will never reach the requested speed or the requested power level'; pick a feed within machine capability so average speed is high. (6) Reduce S-value churn if the engine over-segments — fewer, longer segments per scan line keep the planner cruising. — Applying $32=1 + continuous-emission alone should recover most of the ~24x; M4 + overscan then matches LightBurn on quality (even tone, no corner scorch) at the same or better speed. The investigation should still verify the actual emitted g-code and serial pacing against LightBurn's for the same job — the contract note in the repo (verify external claims against live code) applies: confirm whether the slowdown is firmware config, software-injected stops, or sender back-pressure before patching.

### How LightBurn streams to GRBL and the community-diagnosed causes/fixes for 10x-50x slow GRBL Fill/raster engraves (applied to a 24x-slower-than-LightBurn diode/CO2 GRBL 1.1 fill)
_A 24x gap on a Fill/image engrave at identical image, size, speed and power is almost never the mechanics or the set feed rate — it is the rate at which the host can feed short G-code segments into GRBL and the rate at which GRBL can parse/plan them. LightBurn wins by (a) streaming with the character-counting protocol so GRBL's 127-byte RX buffer and 17-block planner stay full, (b) shrinking command count with GCode clustering (up to ~85% fewer G1 lines for grayscale), (c) requiring laser mode ($32=1) so the machine does not stop on every S-power change, (d) using Fast Whitespace Scan to G0 across blank pixels, and (e) choosing a fill ordering and overscan that keep the head moving at the set speed. The single biggest lever for a 24x case is per-command throughput: 8-bit AVR GRBL parses only ~100-400 motion lines/sec, which caps a 254-DPI fill near ~40 mm/s no matter what feed you set. An app that emits one command per pixel AND streams synchronously (wait-for-ok per line) can land an order of magnitude below LightBurn; combined with $32=0 (3-4x) you reach ~24x. Concrete fixes below, each with quantitative impact and GRBL-1.1 applicability. Hardware note: any g-code-emission/streaming change in LaserForge touches the safety paths in CLAUDE.md (src/communication/, GrblController) and needs failure-mode tests + a Falcon A1 Pro hardware-verification note before being marked Shipped._

- **Character-counting streaming protocol (keep the 127-byte RX buffer + 17-block planner full)**: GRBL exposes two host protocols. Simple send-response sends one G-code line and blocks until 'ok'/'error:' before sending the next; the round-trip lag is small but cumulative and on short fill segments it leaves GRBL's serial read buffer under-utilized and the planner only partially full. The character-counting protocol instead keeps a running tally of bytes sent and only subtracts a line's bytes when its 'ok' returns, so the host keeps stuffing lines until GRBL's 127-character RX buffer is nearly full without overflowing. This 'always ensures GRBL's serial read buffer is filled, while never overflowing it... maximizes GRBL's performance by keeping the look-ahead planner buffer full' (17 line motions). LightBurn's Buffered transfer mode implements exactly this; Synchronous mode is the slow per-line fallback for devices that stutter/restart under buffered. Underrun is the failure being avoided: on an xTool D1 Pro a Python sender that held 32 lines in flight eliminated stutter that appeared above ~105-163 mm/s on the stock synchronous stream. — Removes per-line round-trip stalls; the difference between a starved planner (head decelerates/pauses between segments) and continuous motion. On short fill segments this is the gap between never reaching the set feed rate and reaching it. Accuracy is unaffected (same g-code, just paced differently).
- **Per-command throughput ceiling — the dominant Fill bottleneck (1 command/pixel vs ~100-400 lines/sec on 8-bit GRBL)**: Fill/image engraving emits many tiny moves (often one G1 per pixel/dot). Effective speed is capped by how many motion blocks GRBL can parse + plan + start per second, not by the feed you typed. Empirical/community numbers: 8-bit AVR GRBL ~400 lines/sec (one user measured ~100 lines/sec ≈ ~1000 bytes/sec at his settings); at 254 DPI = 10 dots/mm that caps effective speed near ~40 mm/s (~2400 mm/min) regardless of a 20000 vs 40000 mm/min setting. 32-bit boards are roughly clock-proportional: Arduino Due (84 MHz) ~5x an Uno (16 MHz), reaching ~7000 mm/min raster (step rate only ~9.3 kHz, so steps are not the limit — parsing/planning is). When it slows, 'the planner buffer is starved (not full)' — the MCU can't queue segments fast enough. Higher DPI/shorter intervals make it worse (e.g. 0.5mm→0.5mm spacing dropped a Teensy 4.1 from 18000 to 8569 mm/min). This is the most likely single cause of a 24x gap when one app emits more, shorter commands or paces them more slowly than LightBurn. — This is the order-of-magnitude lever. If an app's fill at 254 DPI is effectively capped ~40 mm/s while LightBurn (via clustering + buffered streaming) sustains the real set speed, that alone is a 5-10x difference; on a heavy-but-mostly-blank image it can exceed 10x. Mitigations: fewer/longer segments, lower DPI/larger line interval, GCode clustering, and a faster MCU. No accuracy loss from streaming faster; loss of detail only if you reduce DPI.
- **GCode clustering (pack up to 16 power values per G1 to beat the command-rate limit)**: Encodes multiple consecutive laser-power (S) values for fixed-spacing sub-segments into a single G1, colon-delimited (e.g. 'G1 Y-0.48 S0.875:0:0:0:0.875:0:0:0.875'), up to 16 values. This collapses many tiny per-pixel commands into one move, so the host sends far less data and GRBL parses far fewer lines while still doing grayscale/diagonal engraving. Real example: a 45x45 mm grayscale at 0.1 mm interval went from 167,526 G-codes (2.27 MB) to 28,975 (1.31 MB) — ~85% fewer commands. It directly attacks the throughput ceiling above and lets the planner see farther ahead. LightBurn negotiates support via the '$I' response containing '[CLUSTER:xx]'. — Reported 4x rastering improvement (>20000 mm/min grayscale at smaller dot) on capable firmware; one Smoothie user 'doubled my speed... from 150 mm/s to 275 mm/s'; an ESP32 deep-grayscale file went 48→33 min (~31%). Eliminates planner-slowdown on dense images. Accuracy preserved (same per-pixel power). CAVEAT: support is firmware-specific — grblHAL, FluidNC, Marlin, Smoothieware and a few aftermarket GRBL boards support it; stock 8-bit gnea/grbl 1.1 generally does NOT, and enabling it there throws a GRBL error. So it is a strong fix for grblHAL/ESP32-class GRBL diode rigs but not for a classic Uno-based GRBL 1.1.
- **Laser mode $32=1 (no stop on every power/S change)**: With $32=0 (spindle mode), GRBL treats every M3/M4/M5 or S-value change like a spindle that must spin up/down: it 'would come to a stop, allow the spindle to change, and then continue.' A fill changes power constantly, so the head stops-and-starts on essentially every segment, producing the classic stutter and a hugely inflated time. $32=1 makes GRBL 'move continuously through consecutive motion commands when programmed with a new S spindle speed (laser power),' i.e. it plans through power changes instead of halting. — Forum-confirmed 3-4x slowdown when wrong ('$32=1 did the trick for me'). On its own this is a plausible large chunk of a 24x case; combined with a per-pixel/synchronous fill it compounds with the throughput ceiling. Also improves edge quality (fewer over-burn dwell points). Must be on for any GRBL diode/CO2 raster.
- **Fast Whitespace Scan (G0 across blank pixels)**: When engraving an image LightBurn normally crosses the whole row at the engrave speed. Fast Whitespace 'boosts the speed through blank areas to the speed you indicate, if it is faster than the current engraving speed' — effectively skipping blank runs at rapid (G0) speed instead of crawling across them lit-but-zero-power. Biggest win on images that are mostly empty. — Savings scale with the fraction of blank space — can be large (a sparse logo on a big canvas may be majority whitespace). No accuracy impact (blank areas aren't burned). Set the whitespace speed to the machine's safe max rate. Applies to any GRBL g-code laser. Recommended explicitly by LightBurn staff for slow jobs.
- **Acceleration ($120/$121) and overscan vs short segments (reaching the set feed rate)**: Effective speed across a short fill line = limited by acceleration: the head ramps up, may never hit the programmed feed, then ramps down. Low accel + short lines means the set speed is fictional. Overscan adds lead-in/lead-out so the head is already at speed when the laser fires (and prevents darker edges). Raising $120/$121 (X/Y accel, mm/s^2) lets the head reach the set speed sooner over short spans. (Note: accel cannot beat the command-rate ceiling above — one user at accel 1500 still couldn't exceed 2500 mm/min because the limit was instruction processing, not mechanics. Diagnose which one binds.) — Where mechanics bind (longer lines, lower DPI), higher accel + overscan can yield a meaningful fraction of the set speed instead of a small one; impact is geometry-dependent (often 1.2-2x on short-line fills). Overscan trades a little extra travel for even burn and full-speed firing. If acceleration is too high for the frame the controller may throw ALARM:2 / out-of-bounds — size overscan to fit. Applies to GRBL 1.1; safety paths note: changing accel affects motion limits.
- **Fill ordering and layer consolidation (scan-all vs fill-shape-by-shape)**: LightBurn's Fill order ('scan all at once' vs 'fill groups/shapes one by one') decides how much the head moves at speed vs how much it stops to reposition. Scanning everything in one pass keeps the head at the set speed for most of the job (best when the laser is fast or accelerates slowly); per-shape filling is better only when accel is fast or the image is mostly blank. An app that fills section-by-section ('1 pixel at a time, moving from one section to another') instead of long continuous scan lines pays the per-segment overhead constantly. Merging layers that share identical settings also removes redundant passes/travel. — Picking scan-all on a slow-accel diode rig can cut large amounts of acceleration/repositioning overhead vs fragmented per-shape fills (commonly 1.5-3x on multi-region art); no accuracy change. Wrong choice is a frequent cause of community 'why is my fill so slow' reports. Applies to any GRBL g-code laser.
- **M4 dynamic power (avoid M3 stop-and-burn artifacts; smoother continuous motion)**: M3 constant power holds the programmed S regardless of speed, so where the head slows for direction changes it over-burns and, with $32=0, encourages stops. M4 dynamic mode 'automatically adjusts laser power based on the current speed relative to the programmed rate' and turns the laser off when not moving, which pairs with laser mode to let GRBL plan continuously through accel/decel without scorching. — Primarily an accuracy/consistency win (even burn through accel/decel) that also enables continuous motion rather than stop-to-change. Indirect speed benefit by removing the incentive to halt at power changes. Requires S-MAX to match $30 for correct scaling. Applies to GRBL 1.1 diode/CO2.
- **Baud rate / link bandwidth (raise serial ceiling when bytes/sec binds)**: If the bottleneck is bytes on the wire rather than MCU parsing, the host can't refill the 127-byte buffer fast enough at high feed on dense images. Stock GRBL is 115200 baud; community reports of raising to 230400/460800 (firmware-rebuild on capable boards) restored higher engrave speeds. WiFi/ESP32 links (>400 kB/s vs ~10 kB/s on old boards) similarly remove the bandwidth wall — xTool's own fix for stutter was 'switch to a Wi-Fi connection.' Relevant to the Falcon A1 Pro WiFi bridge path. — Where serial is the limit, a 2-4x baud increase removes underrun stutter and lets the set feed rate be sustained on dense fills. No effect if the limit is MCU parsing (then use clustering / fewer segments instead). No accuracy impact. Applies to GRBL 1.1 but baud changes require firmware support; treat as a controller-side change, not a host emission change.
- **Line interval / DPI (largest mechanical lever; halve passes by doubling spacing)**: Number of scan rows = engrave height / line interval, so it scales total time linearly. Many slow-fill reports use far finer intervals than the beam needs (e.g. 0.1 mm / 254 LPI when the dot would tolerate 0.2 mm). Doubling the interval to match the actual dot/kerf halves the rows — and, crucially, halves the command count feeding the throughput ceiling above. — Doubling interval ≈ 2x faster ('that'll half your engraving time instantly'); also compounds with the command-rate limit. Accuracy: coarser interval reduces vertical resolution/tonal smoothness, so match it to beam/defocus rather than over-resolving. Applies to all GRBL lasers; the cheapest single fix when DPI is set higher than the optics justify.

### Why a GRBL Fill engrave can run ~24x slower than LightBurn on identical hardware/image/settings: a timing model grounded in serial throughput, GRBL firmware buffering, and CNC motion-planning physics
_A 24x GRBL-vs-LightBurn Fill gap is multiplicative across throughput/firmware/physics limits, not one bug. Grounded in primary sources: GRBL 1.1 has a 128-byte RX buffer (127 usable), a 16-block look-ahead planner (BLOCK_BUFFER_SIZE=16), a 6-entry step-segment buffer, 100 Hz acceleration ticks, and ~30 kHz max step rate on the AVR. Serial at 115200 8N1 caps at 11,520 bytes/s (~520-960 short lines/s ideal), but the real wall is the AVR parse/plan rate — a comparable GRBL-class controller measured ~10 lines/s on an 8-15 lines/mm file, and the GRBL maintainer's raster diagnosis is explicit: 'when a slowdown happens the planner buffer is starved (not full).' The dominant, controllable causes: (1) per-pixel vs per-run move coalescing (lines/mm maps ~linearly to throughput; LightBurn emits long constant-power runs), (2) character-counting streaming vs send-and-wait-for-ok (round-trip latency drains the 16-block buffer), (3) short-segment trapezoidal->triangular collapse where the laser never reaches the programmed feed (v_peak=sqrt(a*d): a 0.2 mm run at a=1000 mm/s^2 peaks at ~14 mm/s vs a 100 mm/s target, ~7x), (4) laser mode $32=1 + M4 to avoid a full stop on every S/power change (a=catastrophic per-pixel stop-and-go when off), plus overscan/fast-whitespace, line interval (DPI), and bi-directional scanning. Enlarging the planner buffer from 36->1024 segments took grblHAL raster from stalling to a smooth 48,000 mm/min, quantifying how much the 16-block buffer throttles short segments. A documented same-hardware report (143 min vs 51 min) confirms order-of-magnitude gaps are software (G-code generation + streaming + whitespace + laser mode), not the machine. The likely 24x = ~3-4x streaming/handshake x ~3-8x per-pixel-vs-per-run/short-segment x possibly ~1.5-2x M3-without-laser-mode._

- **The fundamental bottleneck chain: a Fill engrave is throughput-bound, not feed-rate-bound**: A raster/Fill engrave is decomposed into thousands of tiny G-code moves (one scanline is broken into many short G1 segments with per-pixel S power changes). The end-to-end rate is the MINIMUM of three serial/firmware/physics limits: (1) host->controller serial byte rate, (2) GRBL's per-line parse+plan rate on the AVR, (3) the achievable average velocity given acceleration over short segments. Whichever is smallest sets the dwell-per-line; the laser then moves at that effective rate regardless of the programmed feed. A 24x gap is almost always MULTIPLICATIVE across these mechanisms (e.g. a 3-4x streaming/handshake penalty x a 3-5x short-segment/planner-starvation penalty x a possible 1.5-2x stop-between-moves penalty), not one single cause. The diagnostic signature is that the planner buffer is observed STARVED (not full) during the slow region — the GRBL maintainer states for slow raster: 'when a slowdown happens the planner buffer is starved (not full),' meaning the firmware/serial cannot feed motion fast enough to keep the gantry moving continuously. — Sets the entire job time. If the effective line rate is 60 lines/s vs LightBurn's effective ~1000+ equivalent (via larger/coalesced segments and a tuned controller), the job is ~16-24x longer with NO visible difference in the toolpath — same image, same nominal speed/power. The hardware's rated 20,000 mm/min is irrelevant if the move pipeline can only sustain a few thousand mm/min of effective raster throughput.
- **Serial baud-rate ceiling on lines/second (11,520 bytes/s at 115200 8N1)**: GRBL's default UART is 115200 baud, 8N1 = 10 bits/byte, giving a hard ceiling of 115200/10 = 11,520 bytes/s of usable host->controller throughput. A typical raster G-code line such as 'X12.345S128\n' or 'G1X10.5S255\n' is ~10-14 bytes including the line ending. So the serial link alone caps streaming at roughly 11,520/12 = ~960 lines/s in the ideal case — but only if the line is the minimal short form AND the stream is kept continuously full. Verbose lines (full 'G1 X12.345 Y0.000 S128' with spaces, ~22+ bytes) halve that to ~520 lines/s. This is the THEORETICAL best; real firmware parse limits (next item) are usually hit first. Raising baud (e.g. 230400/250000) roughly doubles this ceiling and a forum report confirms switching 115200->250000 eliminated errors on rapid-heavy files. — At 0.1 mm line interval and short X moves, even the ideal ~960 lines/s ceiling caps effective engrave throughput well below a fast DSP. Verbose/long G-code, low baud, or a slow USB-serial bridge can independently cost 2-4x. Quantitative: dropping from short to verbose line format = ~2x slower; 115200 vs 250000 = ~2x headroom.
- **GRBL per-line parse rate on the 8-bit AVR (~tens to low-hundreds of lines/s for dense raster)**: GRBL on an ATmega328p (16 MHz, 8-bit, no FPU) must read, tokenize, float-parse, and motion-plan each line. For raster files with very short segments this parse+plan cost — not serial bandwidth — is the real wall. A directly comparable data point: a Maslow (also AVR-class GRBL-derived) user measured 'a maximum limit of around 10 lines of gcode per second' on a file with '8-15 lines of gcode per millimeter,' with the maintainer attributing it to 'the arduino reading and parsing.' GRBL's own changelog notes a 'protected planner buffer and new step segment buffer allowed performance increases over 4x' — i.e. the un-optimized parse/segment path was the limiter. The look-ahead planner holds only BLOCK_BUFFER_SIZE = 16 blocks (config.h default) and the step executor a SEGMENT_BUFFER_SIZE = 6 segment buffer; if lines arrive slower than they drain, the planner empties and the gantry is forced to decelerate to a stop at the tail of the buffer. — This is frequently THE dominant term. If GRBL sustains ~60-150 dense raster lines/s while LightBurn+controller effectively sustains the equivalent of many hundreds (by emitting longer coalesced moves the AVR can plan in far fewer lines/mm), that single ratio is ~4-10x. The grblHAL fix of enlarging the planner buffer from 36 to 1024 segments took raster from stalling to a smooth 48,000 mm/min on a Teensy 4.1 — quantifying how much the small 16-block buffer throttles short-segment streaming.
- **Character-counting streaming vs send-and-wait (line-by-line ACK) — keeps the 16-block planner full**: GRBL has a 128-byte serial RX buffer (RX_BUFFER_SIZE=128; 127 usable) feeding the 16-block planner. The recommended host protocol is CHARACTER COUNTING: the sender keeps a running total of bytes in-flight and pushes new lines as soon as <=127 bytes are outstanding, so 'it always ensures Grbl's serial read buffer is filled, while never overflowing it... keeping the look-ahead planner buffer full.' The GRBL wiki states that with this protocol 'seemingly, only the speed of the serial connection is the limit.' The naive alternative — send one line, WAIT for 'ok', then send the next — adds a full USB/serial round-trip latency (often 1-5+ ms each way, worsened by OS serial driver buffering/latency timers) between every line. With short raster lines that round trip can dwarf the move's own execution time, so the planner repeatedly drains to empty and the gantry stutters/decelerates per line. — Huge and very plausibly a large slice of a 24x gap. If each tiny move executes in ~5-15 ms but a send-wait-ok cycle adds ~5-20 ms of idle, effective throughput drops 2-5x AND every short segment also loses its junction speed because the buffer empties (compounding with the short-segment penalty below). LightBurn implements proper buffered/character-counting streaming; a homegrown sender that waits for 'ok' per line is a classic cause of order-of-magnitude raster slowdowns.
- **Short-segment trapezoidal->triangular collapse (acceleration physics: the laser never reaches programmed feed)**: Each move follows a trapezoidal velocity profile: accelerate (distance v^2/2a), cruise, decelerate (v^2/2a). The cruise phase only exists if the segment is longer than s_accel + s_decel. Below that threshold the profile becomes TRIANGULAR — it ramps to a peak v_peak = sqrt(a*d) (for symmetric accel=decel from/to rest) and immediately ramps down, never reaching the programmed feed. Example with GRBL-typical numbers: a = 1000 mm/s^2 (a common $120/$121), target feed = 100 mm/s (6000 mm/min). To reach 100 mm/s needs d = v^2/2a = 100^2/2000 = 5 mm of accel + 5 mm decel = 10 mm minimum for full speed. A raster scanline crossing a 0.2 mm dark pixel run is FAR below this: peak v = sqrt(1000*0.2) = ~14 mm/s = ~850 mm/min — i.e. ~7x below the 6000 mm/min you asked for, purely from physics, even with a perfectly fed buffer. — On detailed images with many short on/off runs and per-pixel feed/power changes that fragment the line, the effective speed is dominated by accel/decel, commonly 3-10x below the nominal feed. LightBurn mitigates this by emitting whole scanlines as longer constant-power runs (fewer, longer segments => more cruise, less ramp) and by adding overscan so the cut portion runs at full speed. Accuracy impact in GRBL non-laser/M3 mode: ends of lines get over-burned because the laser dwells while decelerating — the classic 'dark line ends' symptom.
- **Laser mode $32=1 and M4 dynamic power — avoid a full STOP on every power change**: With laser mode OFF ($32=0), GRBL treats the laser like a spindle: 'Every time a spindle state M3 M4 M5 or spindle speed Sxxx is altered, Grbl would come to a stop, allow the spindle to change, and then continue.' A raster line changes S (power) on essentially every pixel, so $32=0 forces a decelerate-to-zero/accelerate cycle PER PIXEL — catastrophic for fills. With $32=1, GRBL 'move[s] continuously through consecutive motion commands when programmed with a new S spindle speed... updated instantaneously through each motion without stopping.' M4 (dynamic power) additionally scales PWM with instantaneous velocity so power stays correct through the accel/decel ramps and turns the beam off at zero speed. — If a sender ships M3 with $32=0 (or the firmware has laser mode disabled), the per-pixel stop-and-go can slow a fill by 2-10x AND scorch. Enabling $32=1 + M4 is the single biggest 'free' fix and is mandatory for any continuous fill. Quantitatively this alone can recover a multiple-x of a slowdown if it was the cause. Applies directly to GRBL 1.1 diode/CO2.
- **Move coalescing / segment granularity — fewer, longer G-code moves per scanline (the LightBurn advantage)**: The biggest controllable lever for a GRBL sender is HOW the raster is turned into G-code. LightBurn emits a scanline as the minimum set of moves: a single G1 across each contiguous lit run at constant power (or a small number of S-stepped moves), rather than one tiny move per pixel. Fewer lines/mm directly relieves all three bottlenecks at once: less serial traffic, fewer AVR parses, and longer segments that actually reach cruise speed. The Maslow data (8-15 lines/mm => ~10 lines/s) versus a coalesced ~1-2 lines/mm shows the lines/mm ratio maps almost linearly onto throughput. A naive engraver that emits per-pixel X+S moves at, say, 0.1 mm interval generates 10 lines/mm; coalescing dark runs can cut that 5-10x. — Often the decisive multiplier. Cutting lines/mm by ~8x (per-pixel -> per-run) is roughly an 8x throughput win on a parse-bound AVR and simultaneously lets segments reach full feed. This is the most likely single explanation for a 24x gap if the slow app emits per-pixel moves while LightBurn emits per-run moves.
- **Overscan / fast-whitespace / scanline skipping — don't crawl the blank margins**: Overscan extends each scanline past the image edge so accel/decel happens with the laser OFF, and the lit portion is traversed entirely at full programmed speed (LightBurn docs: lets the laser 'accelerate and decelerate with the laser off... only fires once at full speed'). 'Fast Whitespace' traverses blank regions at a higher rate than the engrave rate (LightBurn dev: 'traversing the very large blank areas between cuts is done at a different rate than the cutting itself'). Skipping fully-blank scanlines avoids emitting moves for empty rows entirely. — For sparse/line-art-on-white images this is large — blank rows and margins can be most of the bounding box. Fast-whitespace + blank-line skip can cut total time 2-5x on images with lots of background, and overscan removes the end-of-line dwell that both slows and over-burns. A sender that engraves every row edge-to-edge at the slow engrave feed pays full price for empty space.
- **Line interval (DPI) — the scanline-count multiplier on total time**: Total scanlines = engrave height / line interval, so time scales ~inversely with line interval. Halving the interval (e.g. 0.1 mm -> 0.05 mm, ~254 -> ~508 DPI) doubles the number of passes and roughly doubles job time. LightBurn docs: 'If Line Interval is set too low, the engraving time is vastly increased.' If the slow app silently uses a finer interval (or the two apps interpret DPI differently), that is a direct 2x+ on its own. — Linear in time. A 2x finer interval = ~2x longer. Worth confirming both apps use the identical interval before attributing the gap to firmware/streaming; a mismatched interval is an easy hidden multiplier inside a 24x total.
- **Bi-directional scanning and inside-out/path ordering — halve travel and kill rapids**: Bi-directional (zig-zag) raster engraves on both left->right and right->left passes; uni-directional wastes one whole traverse per line returning with the laser off. LightBurn docs: bi-directional 'is usually preferred because it is twice as fast.' (Trade-off: needs Scanning Offset Adjustment to correct the per-direction PWM/firing latency that otherwise shifts alternate rows.) Good scan ordering also minimizes rapid repositioning between fill regions. — Uni- vs bi-directional is up to ~2x on the engrave portion alone. Combined with fast-whitespace it removes most of the non-cutting time.
- **PWM update / step-pulse timing on the AVR (a secondary ceiling, usually not the 24x cause)**: GRBL on the 328p tops out near a 30 kHz step rate and updates spindle PWM inside the stepper ISR. The grblHAL discussion found that for 7000 mm/min the step pulse freq was only ~9.33 kHz (NOT the bottleneck) and suspected 'PWM update time' as a finer constraint at very high speeds. Acceleration is recomputed at ACCELERATION_TICKS_PER_SECOND = 100 Hz. These cap the TOP achievable raster speed but are rarely what makes a job 24x slower — they bound the ceiling, not the typical short-segment-starved operating point. — Bounds maximum sustainable raster speed (tens of thousands of mm/min on AVR is unrealistic; high-rate raster wants a 32-bit controller). Minor relative to streaming/segment effects for a 2h-vs-5min gap; mention for completeness and to rule it out.
- **Real same-hardware evidence that the gap is software, not the machine**: Multiple users report large job-time differences on identical hardware/settings driven only by the sender software. A LightBurn forum report: same machine, same 1500 mm/min and power, LightBurn = 143 min at 0.133 mm interval vs LaserGRBL = 51 min — and the LightBurn dev attributed closing such gaps to enabling Fast Whitespace and proper path handling. The direction can go either way depending on settings, but the key point for diagnosis is that a 2-3x (and by extension a 24x) difference is fully explainable by streaming protocol + segment generation + whitespace handling + laser-mode, with the gantry never changing. — Confirms the model: target the software's G-code generation and streaming, not the firmware/hardware. Establishes that order-of-magnitude software-only gaps are real and documented.

## Relevant decisions (from your DECISIONS.md / code)

- FILL GEOMETRY — compile-time, not emit-time (ADR-019, Accepted 2026-05-28). When a layer's mode is 'fill', hatch-line generation happens in src/core/job/compile-job.ts via fillHatching(), NOT in the GRBL strategy. A closed Polyline is replaced by parallel scanline hatch lines (even-odd fill rule → handles holes like letter 'O'; open polylines silently skipped). Two Layer fields: hatchAngleDeg (0..180, default 0) and hatchSpacingMm (default 0.2mm ≈ 5 lines/mm). Deliberately rejected pushing fill into grbl-strategy (would duplicate scanline math per OutputStrategy) and a separate FilledRegion SceneObject (every CAM tool keeps Fill as a layer attribute). Self-implemented ~150 LOC; no maintained MIT lib (flatten-svg/polygon-fill) covered the closed-polyline+angle+spacing case.

- FILL geometry implementation (src/core/job/fill-hatching.ts) is snake/serpentine by design: pushScanlineHatches alternates row direction by scanIndex parity (forward = scanIndex%2===0) so the laser avoids a return-to-start travel between rows — this is a deliberate speed optimization documented in the file header step 4. Angle handled by rotating geometry into a horizontal-scanline frame and back. Half-open [yLo,yHi) interval avoids vertex double-count; snapTrig snaps cos/sin near 0/±1 to kill a spurious-row 'lip' at right angles. MIN_HATCH_SPACING_MM clamp = 0.05mm guards against an effectively-infinite loop / enormous G-code.

- FILL PERFORMANCE — a known-and-fixed app-freeze. fill-hatching.ts was rewritten from O(scanlines × edges) (which froze the whole app when a traced 'big image' of thousands of contours was switched to Fill, because canvas redraw AND live ETA both run it synchronously) to a single-pass active-edge sweep: buildSortedEdges() once, advancing cursor admits edges by yLo, half-open y<yHi retires them, so cost tracks geometry not bed height. Guarded by src/core/job/fill-hatching.perf.test.ts: 35,000 contours must fill in <5000ms (naive ~9s; active-edge ~0.4s isolated, up to ~1.8s under suite CPU contention). Additionally memoized: src/core/job/fill-hatching-cache.ts caches hatch results in a WeakMap keyed by the polyline array, sub-keyed by 'angle:spacing' (LRU cap 8 settings) so pan/zoom/scrub redraws don't recompute hatches.

- M3 vs M4 POWER — hardcoded per layer.mode, no UI/profile knob (ADR-020 Q1, Accepted 2026-05-28). Image/raster groups emit M4 (dynamic — power scales with feed); line AND fill groups emit M3 (constant). Rationale: the M3/M4 choice is dictated by what the laser is doing (cut/engrave vs raster), not operator preference; exposing it invites misconfiguration. Escape hatch deferred: add a DeviceProfile forceMMode field only if a controller variant later needs M3 for raster.

- M3/M4 implementation confirmed in code. src/core/output/grbl-strategy.ts: preamble() emits 'G21 / G90 / M3 S0' (M3 S0 primes the controller so a later G1 S>0 fires even when not in laser mode $32=0). emitGroup (cut) and emitFillGroup both stay on that M3 — Fill is NOT flipped to M4. Raster is dispatched to src/core/raster/emit-raster.ts (emitRasterGroup), which emits 'M5' then 'M4 S0' at group start and a trailing 'M5'. emitJob re-issues 'M3 S0' when a cut/fill group follows a raster group, so mode state is never left dirty across groups.

- M4 FILL was deliberately NOT adopted in the overscan increment (ADR-031 decision #4 + RESEARCH_LOG 'Fill hatch overscan' note). LightBurn's GRBL default is M4 (Variable Power) with M3 (Constant Power) as a compatibility option, and the research confirms this — but LaserForge kept Fill on M3 because switching Fill M3→M4 also changes cut-depth behavior on short hatches and enlarges the diff blast radius. M4 Fill is explicitly parked as a separate hardware experiment if edge marks remain after overscan. This is a candidate root cause if Fill burns look wrong vs LightBurn (LightBurn would be on M4).

- OVERSCAN for Fill (ADR-031, Accepted 2026-06-01). Root cause it fixed: the first real logo burn showed darker marks at the sides of filled regions — the standard endpoint-overburn failure where hatch lines start/stop at the object boundary while the gantry is still accelerating/decelerating. Decision: added a distinct FillGroup to Job (kind:'fill' vs cut's kind:'cut'); added Layer.fillOverscanMm (default 5mm, back-filled from LAYER_DEFAULTS, no schema bump); each 2-point hatch expands to lead-in/burn/burn/lead-out with S0 on the runway (G0 leadStart S0 → G1 burnStart F S0 → G1 burnEnd Spositive → G1 leadEnd S0). Positive power only ever rides a moving G1, never a stationary modal command. Implemented in src/core/job/fill-overscan.ts (expandFillHatchWithOverscan) consumed by grbl-strategy.emitFillGroup. KNOWN TRADEOFF stated in the ADR: 'More overscan increases runtime' and jobs near a bed edge can now fail preflight (correct — overscan is runway, not artwork; Frame still traces only the burn area).

- BIDIRECTIONAL RASTER — a regression FIX, and the key declared performance story (ADR-032, Accepted 2026-06-01). After ADR-031 overscan fixed the side burn marks, the next hardware observation was that a print took ~3× longer. Cause: overscanned raster was still UNIDIRECTIONAL — every row burned left-to-right, exited the right overscan zone, then wasted a full return move to the left overscan before the next row. Decision: keep overscan; change ONLY raster/Image row emission to serpentine (emitted active row 0 L→R, row 1 R→L, …); alternate by EMITTED active-row count, not raw pixel-row index, so skipped blank rows don't force a long return sweep; preserve active-span clipping, M4, and S0 runway both ends; NO UI toggle in this patch (bidirectional ghosting/backlash, if seen at high speed, becomes a later unidirectional/scanning-offset-calibration option). Note: Fill hatches were already serpentine in fill-hatching.ts; the regression target was raster only.

- BIDIRECTIONAL raster implementation confirmed in src/core/raster/emit-raster.ts. The main loop tracks emittedSweepCount and passes reverse = (emittedSweepCount % 2 === 1) to emitRow; reverse rows swap start/end X (activeEnd+overscan → activeStart−overscan) and walk runs right-to-left (emitReverseRowRuns) vs left-to-right (emitForwardRowRuns). Three stacked speed optimizations are documented inline: (1) skip all-S0 rows entirely (a 200×100mm banner 80% blank drops ~5×), (2) clip each row's sweep to its active span [first non-zero .. last non-zero] + overscan, (3) serpentine to avoid the full-width return. Covered by emit-raster.test.ts:153 ('alternates active raster rows so blank rows do not force return sweeps') and :261 ('emits reverse-row S runs from right edge to left edge'). Also: run-length compression — a G1 is emitted only when dithered S changes (modal F on the very first G1 only, modal S only on change).

- DITHERING — roll-our-own, three modes (ADR-020 Q2). src/core/raster/dither.ts implements threshold (single cutoff), floyd-steinberg (greyscale error-diffusion, serpentine scan), grayscale (direct luma→S, no dithering, for 256-level-S lasers). floyd-steinberg npm pkg rejected (last published 2017 → fails ADR-017 maintenance check); algorithm is ~30 LOC of standard math.

- STREAMING — character-counted buffer, pure state machine (ADR-004 phased delivery; impl in src/core/controllers/grbl/streamer.ts). GRBL RX buffer treated as 120 bytes (DEFAULT_RX_BUFFER_BYTES) — deliberately reduced from a prior 127 (1-byte margin, off-by-one in the safe direction) to match CNCjs's 120 (8-byte safety margin against senders that add CR/LF); 'No observed bug at 127 — preventive' per MIT-compare audit. step() sends as many queued lines as fit in remaining headroom; onAck() pops head-of-queue on each ok/error/alarm and frees its bytes (alarm → status 'cancelled'). Module is pure — it answers 'what to send next', the platform adapter does byte I/O. StreamerStatus distinguishes 'cancelled' (user stop) from 'disconnected' (cable yank) for correct UI wording, both terminal (CNCjs parity, MIT-T1 audit).

- RASTER STREAMING THRESHOLD — declared but NOT yet implemented (ADR-020 Q3). Decision: emit a full string up to 100KB, switch to a generator/AsyncIterable past that (50×50mm photo @5 lines/mm ≈ 7.5KB fits in memory; 200×200mm @10 lines/mm ≈ 6MB must stream); 100KB matches the AUDIT A6 bundle-split heuristic. HOWEVER emit-raster.ts header explicitly lists 'Async-iterable emit for >100 KB jobs (ADR-020 Q3 threshold)' under 'Future improvements (out of F.2.b v1)' and emitRasterGroup currently returns a complete string only. So large-raster memory pressure is a KNOWN, deliberately-deferred gap, not yet wired.

- STREAMING SAFETY race fixes in the store (src/ui/state/laser-store.ts). resumeJob and stopJob use Zustand functional set() so the streamer snapshot is taken AT WRITE TIME — a plain `const s = get().streamer` before set() could clobber concurrent ack-driven advanceStream updates and drift in-flight byte accounting against the real GRBL buffer; the comment warns 'accounting drift can push more bytes than the buffer holds → dropped commands → uncontrolled head motion' (R-H2 audit). onClose marks a streaming/paused streamer 'disconnected' rather than leaving stale 'streaming'.

- STATUS POLLING throttle — a deliberate CPU/serial-chatter optimization (laser-store.ts). Polls at a fixed 250ms tick (STATUS_POLL_MS) but emits the '?' status request only every 4th tick (IDLE_POLL_DIVISOR=4 → effective 1000ms) when NO job is active; keeps the full 250ms (~4Hz) cadence while streaming/paused so live progress stays smooth. Rationale: CNCjs/LightBurn both ramp polling down when idle (MIT-T2 audit). WCO is only reported by GRBL on a cadence, so the store caches the last non-null WCO in wcoCache (UI reads wcoCache, never statusReport.wco); cleared on disconnect/alarm/soft-reset to match GRBL clearing G92.

- DECLARED PERFORMANCE NON-NEGOTIABLES (PROJECT.md 'Success metrics' / 'Accessibility / performance'): 60 fps pan/zoom on a 5,000-segment scene; G-code generation < 2s @ 5,000 segments; web cold-start < 2s, desktop cold-start < 3s; web bundle < 1MB compressed (each dependency counts against budget); fresh-install user → physical cut piece in < 10 min. Determinism non-negotiable #5: same input + params → byte-identical G-code (snapshot-tested) — this constrains every emit change. Non-negotiable #3: every G0 carries S0 / precedes M5 (laser-off-on-travel, property-tested) — enforced in grbl-strategy and emit-raster.

- KNOWN PERFORMANCE ISSUE — post-trace 'unusably slow' workspace (RESEARCH_LOG 'Workspace image-scan responsiveness', 2026-06-01). User reported the app froze after applying an image trace/scan even though trace geometry was acceptable. Fixes shipped: draw-vector-strokes.ts/new display-polylines.ts bound and cache the display sample by immutable polyline array (was reading every source point); draw-preview.ts had been compiling the job + rebuilding the toolpath INSIDE the draw function — Workspace.tsx now memoizes the preview toolpath by `project` while preview mode is active so pan/zoom/scrub don't recompile; worker fallback bounded (large images error instead of silently tracing on the main thread). Decision: keep full geometry for saved projects/G-code/Start, simplify ONLY display/preview rendering. OffscreenCanvas + transferable ArrayBuffers explicitly parked as future work if profiling still shows long tasks.

- KNOWN GEOMETRY-QUALITY GAP (not perf) — trace is outline-only (PROJECT.md Phase E note). imagetracerjs produces outline contours, so a single pen stroke becomes TWO parallel contours; closing this outline-vs-centerline gap (a centerline/skeleton trace mode) is called out as 'the core remaining faulty-vs-LightBurn issue' and is NOT caught by the IoU perceptual harness. Relevant if Fill of a traced stroke looks doubled/wrong vs LightBurn — it's an upstream trace-geometry issue, not a Fill or emit bug.

- EVIDENCE-DISCIPLINE caveat for the diagnosis: ADR-031/032 confidence is 'high for overscan root cause; medium' on exact overscan default and 'medium on hardware backlash risk until the user burns a bidirectional test'. The serpentine raster fix and Fill overscan are code-shipped + unit/property-tested but several items carry Falcon hardware-verification-pending status (ADR-020 F.2.f photo burn, ADR-031 0/2/5mm overscan sample, ADR-032 bidirectional ghosting). So 'we chose serpentine and it should recover ~3× runtime' is a tested code claim, but the on-metal speed/quality outcome was not yet confirmed in these docs.
